Abstract
Provenance refers to information about the origins of objects or data,
including how, by whom, and through what processes it was produced.
The Common Provenance Model (CPM) is a standard for representing
provenance information, allowing the distribution of this information
among organisations.
The main focus of this work is the implementation of a library for
the creation and manipulation of individual provenance documents
in the CPM form, developed as an extension of an existing library
for working with provenance, ProvToolBox. The library represents
the individual provenance documents in-memory as a traversable
graph, enables their full editability and offers numerous quality-of-life
functionalities for working with CPM. Furthermore, the library allows
the creation of the provenance documents in the CPM form using a
templating mechanism.
Additionally, the library is demonstrated using two datasets provided by Masaryk Memorial Cancer Insitute (MMCI) and European
Marine Biology Resource Centre (EMBRC), respectively.
Keywords
provenance, W3C PROV, Common Provenance Model, PROV Template, ProvToolBox

Introduction
In an evermore interconnected world, with printed documents on the
decline for decades, a need arises to store and share information about
research objects, such as datasets, digitally between organisations in a
common, secure, and reliable way.
One approach to capturing such data is by using the concept of
provenance, which is the information about history of objects or data.
For the purpose of representing this information, the PROV Data
Model [1] has been specified.
However, this model by itself does not specifically support the distribution of this information between multiple facilities. Therefore, a
Common Provenance Framework is currently under development [2],
serving as a basis for the ISO 23494 standard. The framework specifies
a Common Provenance Model (CPM), which is an extension of the
PROV Data Model. This thesis serves as a reference implementation
of the current state of this model through the development of a new
library. The main concerns of the library include the generation of
provenance documents using a templating mechanism, manipulation
of these documents and their traversability in-memory.
Chapter 1 provides an overview of all relevant background concepts alongside descriptions of two of the most popular provenance
libraries. Chapter 2 describes in detail the requirements placed on
the reference implementation and its design. Chapter 3 provides the
structure of the library and implementation details.
Chapter 4 describes the application of the library on data provided
by Masaryk Memorial Cancer Insitute (MMCI) and a comparison of
templating mechanisms used previously and the mechanism developed in this work. Chapter 5 describes the application of the library on
provenance metadata regarding datasets provided by the European
Marine Biology Resource Centre (EMBRC).

1 Background and Fundamental Concepts
In this chapter, the basic concepts relevant to the work are discussed.
First, the PROV Data Model itself, its different representations, and a
provenance generation mechanism using templates will be described.
Then, the extension, the Common Provenance Model, will be described, followed by a brief description of existing libraries for working
with provenance, ProvToolBox and the Python PROV package.
1.1 The PROV Data Model
Provenance, as specified by W3C [1], is information about activities
and entities that influence an object or a piece of information. The
PROV Data Model [1] is a generic data model that is used to describe
the provenance of objects and information in a standardized form that
can be exchanged between different systems.
An instance of the model, more specifically a package of PROV
structures and namespaces, is referred to as a document, as defined in
the PROV Notation specification [3], which is one of the representations of the PROV Data Model. The concept of a document is not part of
the PROV Data Model specification [1], but as it is also present in the
definitions of other formats, it will henceforth be used in this work.
1.1.1 Core structures
The core structures of the PROV Data Model [1] are of two categories:
types and relations between them. The types are described as follows:
• Entity - A physical, digital or other kind of an object.
• Activity - A process that occurs over time and works with
entities.
• Agent - An organization, individual, or other agent responsible
for activities or existence of entities.
Every relation has a source and a target. The specific core relations
are described below, with the source type listed first and the target
type second.

• Usage - An Activity used an Entity.
• Generation - An Entity was generated by an Activity.
• Derivation - An Entity was derived from another Entity.
• Association - An Activity was associated with an Agent.
• Attribution - An Entity was attributed to an Agent.
• Communication - An Activity was informed by another Activity.
• Delegation - An Agent acted on behalf of another Agent.
The PROV data model can be represented visually in the form of a
directed graph, where the type instances are nodes and the relations
are edges. The entities are graphically represented as ovals, the activities are represented as rectangles, and the agents are represented as
pentagons1
. The relations are represented as directed edges pointed
at the target of the relation.

1.1.2 Identifiers
Identifiers in the PROV Data Model are references used to distinguish
different structures. The identifiers are in the form of qualified names.
A qualified name consists of two parts: a namespace and a local part
denoting the actual identifier. A namespace, which is an IRI, can be
substituted by a prefix, which is declared at the start of a document. If
no prefix or namespace is part of an identifier, the default namespace is
used, also defined at the start of a document. Identifiers are mandatory
for types, and optional for relations, with the exception of Alternate
and Specialization relations, defined in Chapter 1.1.4, which do not
have an identifier.
1.1.3 Attributes
Every structure in the PROV Data Model can have attributes, with
the exception of Alternate and Specialization relations. An attribute
consists of a qualified name that denotes the type of the attribute and
a value. A structure can have multiple attribute values denoted by the
same qualified name. A value can be a qualified name or any RDFcompatible data type, defined in the RDF specification [4]. The PROV
Data Model contains multiple predefined attributes, some of which are
allowed only in certain structures. These attributes are: prov:label,
prov:location, prov:role, prov:type and prov:value.
1.1.4 Extended structures
In addition to the core structures, the PROV Data Model specification [1], defines several relations and sub-types that are considered
extended structures.
Expanded relations
The specific expanded relations are described as follows, with the
source type listed first, and the target type second:
• Start - An Activity was started by an Entity.
• End - An Activity was ended by an Entity.
4
1. Background and Fundamental Concepts
• Invalidation - An Entity was invalidated by an Activity.
• Alternate - An Entity is an alternate of another Entity.
• Specialization - An Entity is a specialization of another Entity.
The expanded structures also define a super-type for all relations,
described as:
• Influence - An influencee was influenced by an influencer.
Every relation, whether part of the core structures or the extended
structures, is also considered an influence. It is recommended by
W3C [1] to use specific relations between types instead of the generic
influence.
Subtypes
Sub-typing can be applied to core structures, in order to define their
specific characteristics more clearly. The subtype is denoted as a prov:
type attribute with a specific type. For Entities, these subtypes exists:
• Plan - An Entity that represents a set of actions.
• Bundle - See Bundles for details.
• Collection and EmptyCollection - See Collections for details.
For Activities, these subtypes exists:
• Revision - An Entity was a revision of another Entity, subtype
of Derivation.
• Quotation - An Entity was quoted from another Entity, subtype
of Derivation.
• PrimarySource - An Entity had a primary source in other Entity,
subtype of Derivation.

For Agents, these subtypes exists:
• SoftwareAgent - An Agent that is a software program, application, etc.
• Organisation - An Agent that is a company, institution, facility,
etc.
• Person – An Agent that is an individual human.
Bundles
A special type of extended structure is a Bundle, which is a subtype
of entity. It is a named collection of structures that allows grouping
them under a single identifier. Bundles may include their own namespace declaration, which extends the declaration at the beginning of a
document. Nesting of bundles within other bundles is not allowed.
Collections
Another special type of extended structure worth describing separately
is a Collection. Collection is a subtype of entity, and consists of other
entities. If an Entity is a collection, but does not have any members, the
subtype of the entity can be EmptyCollection. The relation between
collection and its contents is an expanded n-ary relation, defined as:
• Membership - An Entity had entities as members.
1.1.5 Constraints
The PROV Constrains, specified by W3C [5], are a set of rules to
ensure consistency and validity of a model instance. Their purpose is
to mitigate errors and enforce logical relations in PROV documents.
Some important constraints include:
• Order constraints: All structures must follow logical time order,
for example an end cannot occur before a start.
• Type constraints: Structures must be of correct types, for example relations must be between types of the correct type.
6
1. Background and Fundamental Concepts
• Key and Uniqueness constraints: Constraints 22 and 23 define
that an identifier is a key for a type or a relation, but this does
not mean that the same identifiers cannot be used in multiple
structures. According to Constraint 53, no two relations can
share the same identifier unless they are of the same type of
relation. Constraint 54 further specifies that the identifiers of
types cannot be used as identifiers for relations. However, as per
Constraint 55, two types can share the same identifier, with the
only exception being that an entity and an activity cannot share
an identifier. If two types or relations with the same identifier
are of the same type of structure, a merging algorithm is applied
to decide whether the structure is valid. The merging can fail if
uniqueness constraints, such as the uniqueness of a start time
in an activity, are violated.
A normal form is the set of provenance statements resulting from applying all definitions, inferences, and uniqueness constraints [5]. A document
is considered valid if all constraints are satisfied on the normal form.
1.2 The PROV Data Model Representations
The PROV Data Model can be represented in various data formats,
including XML [6], JSON [7], JSON-LD [8], in addition to its own
format - the Provenance Notation (PROV-N).
1.2.1 The Provenance Notation
The PROV-N data format, as specified by W3C [3], is aimed to be a
human readable representation of the PROV data model. Each document begins with a declaration of namespaces, specifying the prefix
and IRI for each using the line:
prefix ex <http://example.org/>
The document then contains structures in the form of
structure(id1, ..., idn, [a1 = ’v1’ %% t1, ...])
where id1, ..., idn are the mandatory and optional identifiers
for the structure. Every structure has a prescribed order of the mandatory and optional identifiers. The absence of an optional identifier

can be denoted with - sign. The a1 =’v1’ %% t1, ... denotes the
attributes of the structure in the form of qualified names, their values
and the types of those values, respectively. If the value is a qualified
name, than the type can be omitted. If a structure does not contain
any attributes, the attribute list can be omitted altogether.
Bundles are in the form of
bundle ex:identifier
prefix ex <http://example.org/>
...
structure(id1, ..., idn, [a1 = ’v1’ %% t1, ...])
...
endbundle
with namespace declarations first, then a list of structures following.
1.2.2 JSON-LD
JSON for Linked Data, specified by W3C [9], is a format for integration
with semantic web technologies, representing interconnected data
through the concept of a context and IDs. By using IDs, other objects,
not necessarily present in the file, can be directly linked. A context
defines the mapping of terms or prefixes to IRIs. If a document is
without a context, it has every IRI in it’s absolute form.
JSON-LD specifies multiple processing algorithms that can be
applied on any file:
• Compact - Compacts a file using a specified context. Can be
used to switch between contexts.
• Expand - Converts a compact file into its full form by removing
the context.
• Flatten - Flattens nested nodes in a file, and assigns identifiers
to nodes that do not have them.
• Frame - Reshapes a file based on a template.
The PROV Data Model can be serialised in the JSON-LD format.
W3C has specified a JSON schema [8] for JSON-LD serialization and
also a context. Notably, attribute values are represented as arrays, and
the schema does not allow nested structures within structures, with
the exception of bundles.

1.3 The PROV Template
The PROV Template [10] is a templating mechanism for generating
provenance documents. It requires two files, the template itself in
PROV-N format, specifying the topology of the document, and a set
of bindings in JSON format, consisting of the concrete values to the
template’s variables. During the combination of the two files, called
expansion, the variables in the template are replaced by the corresponding values from the bindings file, resulting in the final PROV
document.
In Listing 1.1 an example of a template is provided. The template
contains qualified names with the prefix var. These are the variables
of the template that will be replaced by the values in the bindings file.
document

prefix var <http://openprovenance.org/var#>
prefix estat <http://purl.org/net/statjr/ns#>

bundle var:bundle
activity(var:block_instance, [t:startTime=’var:starttime’, t:endTime=’var:endtime’, prov:type
=’var:block_type’, t:label=’var:block_title’, estatwf:block=’ var:block_uri’])
activity(var:parent)
agent(var:agent)
wasAttributedTo(var:parent, var:agent)
wasStartedBy(var:block_instance, −, var:parent, −, [t:time=’var:starttime’])
entity(var:consumed)
used(var:block_instance, var:consumed, [t:time=’var:consumed_at’, estat:bindingname=’
var:consumed_name’])

1.4 Common Provenance Model
The Common Provenance Model, developed through collaboration
between various institutions [2], is an extension of the PROV Data
Model. It defines several new subtypes of structures, along with multiple additional attributes. The main motivation behind the creation
of the model is to support advanced reliability and traceability of
information to enhance reproducibility, trustworthiness, and quality
assessment of research objects, such as biological samples or derived
data that are documented by provenance.
When a research object or its derivatives are shared among different organizations, each party can only offer provenance information
for the part of the object’s lifecycle they handled. The object is seen as
10
1. Background and Fundamental Concepts
an output from the sender’s process and as an input into the receiver’s.
Each segment of provenance is packaged into a provenance component, realised as a PROV bundle. Connecting these components creates
potentially distributed provenance chains. The main goal is to support
backward traceability so that organisations can verify the source and
history of an object, but forward traceability is also supported.
The core concept of the CPM is the conceptual separation of a
bundle into two parts:
• The traversal information (TI) - Contains forward and backward traversal information and highlights the core process detailed in the bundle.
• Domain-Specific Provenance (DS) - Contains additional information about the process detailed in the bundle.
CPM does not explicitly define how many bundles can be present in
a PROV document. The traversal information follows a standardized
form prescribed by CPM, utilizing concrete structure subtypes, while
the domain-specific part adheres only to PROV-DM, allowing for more
verbosity depending on the process that’s represented.
1.4.1 Subtypes
The new subtypes, defined by the CPM specification [2], are all part
of the traversal information of a bundle. The subtypes, denoted with
the prov:type attribute, are described as follows:
• Main Activity (MA) - The core activity represented by the
bundle.
• Backward Connector - An entity, representing the object that
has been received, e.g. the input of the MA.
• Forward Connector - An entity, representing the object that
can be or has been sent to a receiver, e.g. the output of the MA.
• Sender Agent - An agent representing the sender of the object
between provenance components.
11
1. Background and Fundamental Concepts
• Receiver Agent - An agent representing the receiver of the
object between provenance components.
• Identifier Entity - An entity representing an external identifier.
Figure 1.2: Traversal Information part of a CPM bundle, taken from [2]
The relations between these subtypes can be seen in Figure 1.2.
There is exactly one main activity in a bundle, which can use a backward connector and generates a forward connector. The forward connector, found in the sender’s provenance, links to the next bundle in
the provenance chain and shares its identifier, carrying the sender’s
prefix, with the corresponding backward connector in the receiver’s
provenance.
The backward connector contains additional attributes, such as
hash or bundle id of the previous bundle, allowing backward traversability. The CPM expects all the information for backward traversal
to be present at the creation of a bundle. A specialisation of a forward
connector can also be present in the bundle, containing additional
attributes referring to the next bundle, allowing forward traversability.
Additionally, CPM allows multiple backward or forward connectors to be present in one bundle, connected via derivations, enabling
deeper traversal through the provenance chain, or branching.
The traversal information and the domain-specific provenance are
connected via specializations of the connectors, as can be seen in Figure 1.3, which is an example of the first bundle in a provenance chain.

1.4.2 Versioning
Once a CPM bundle is created, it is considered immutable, as per
the CPM specification [2]. This means the original version is always
preserved without modifications. However, since some information

1.5.1 Model Design
The in-memory representation of the PROV documents is based on the
composition over inheritance principle in OOP [12, p. 20]. This means
that instead of inheritance from parent classes, there are multiple interfaces defined for each common characteristics between structures.
The interfaces for all PROV structures and attributes, including the
Document interface, are defined in the prov-model module. The concrete objects, based on the implementation chosen, are instantiated by
the ProvFactory class, following the Abstract Factory design pattern.
The IndexedDocument class, which supports querying and merging
of documents, is part of the prov-model module. It is an implementation of the StatementAction interface, which prescribes an action for
every Kind of Statement. The constructor of the class takes a Document
and indexes each Statement based on its Kind and, if present, its identifier. The representation is mutable and allows for a transformation
back into a Document, but even if no changes were made, the order of
the statements is not preserved.
1.5.2 Vanilla Implementation
The library includes a basic implementation of the model’s interfaces,
located in the vanilla package of the prov-model module. In this
vanilla implementation, each Kind of Statement does not contain reference to other Statements that utilize its qualified name. For example,
relations do not contain reference to the elements they connect. As a
result, the implementation can process partially complete documents
that lack the expected elements, because it does not verify whether
an element with the correct qualified name exists nor if it is of the
correct Kind. The implementation of a Document is a linked list of
Statements and Bundles, which are also linked lists of Statements
and is not indexed at all. In addition to the vanilla Java implementation
of the model, the library also contains a mutable and immutable Scala
implementation.

2.1 CPM reference implementation requirements

5. Singular template for traversal information instantiation.
6. Traversability and queryability of the bundle.

6.1 Retrieve entity, agent or activity from an identifier.
6.2 Retrieve relation from an identifier or from source and target

6.3 Retrieve main activity.
6.4 Retrieve forward connectors.

2.2 Analysis of approaches
As the requirements state, the reference implementation shall support
the full verbosity of the PROV Data Model, in addition to serialisation
and deserialisation into common PROV formats. These requirements
are non-trivial, hence the sensible approach was to consider extension
of existing libraries that already implement this functionality so that
the CPM-specific requirements can be the focus of this work.
Previous attempts have been made to represent PROV documents
as graphs. One such attempt was a Python library, developed as a
Bachelor thesis by Roland Olejník [14]. Although the solution achieved
its goals, the library lacks serialisation and deserialisation capabilities,
and as such, it was not a valid choice for this work.
Out of the existing state-of-the-art solutions, the ProvToolBox
developed in Java [11] and the Python Prov package [13], it has
been decided to extend the ProvToolBox. The cleaner overall architecture of the library, allowing for greater extensibility, the queryable
IndexedDocument class that can serve as inspiration for the design,
the reference implementation of the PROV Templates, along with

greater personal experience with the Java language itself, were all
reasons for the final choice. Henceforth, ProvToolBox terms, defined
in Chapter 1.5, will be used to refer to PROV terms in this work.
2.3 Design of the extension
By choosing to extend the ProvToolBox library, requirements no. 1
through no. 3 have been inherently satisfied. To satisfy requirements
no. 4 and no. 6, IndexedDocument class served as inspiration. A CpmDocument class has been created implementing the StatementAction
interface and supports conversion from Document and back. The CpmDocument class represents a PROV document with one CPM bundle.
2.3.1 Graph representation
The internal implementation in the CpmDocument is a traversable graph.
As such, references in elements and relations must be included. Creating a new implementation for each Statement was considered, but it
was deemed excessive and was ultimately not needed. Instead, two
wrapper interfaces have been created, utilizing the Decorator pattern
and conforming to the composition over inheritance principle.
An INode interface has been defined that wraps all elements of the
same Kind with the same identifier, satisfying identifier constraints,
explained in Chapter 1.1.5. A handleDuplicate method has been defined, to allow separate handling of duplicate elements in each implementation of the interface. A node also contains edges in which the
node acts as either a cause or an effect.
Furthermore, an IEdge interface has been defined that wraps all relations of the same Kind with the same cause and effect. A handleDuplicate method has been defined for the same purpose as in the INode
interface. An edge contains a cause node and an effect node.
The INode and IEdge both extend a Component interface, uniting
both as a component of the in-memory graph. Lastly, an ICpmFactory
interface has been defined that wraps ProvFactory interface to instantiate concrete implementations of the INode and IEdge interfaces,
supporting different internal representations. The class diagram in
Figure 2.1, shows the described interfaces.

3.2.1 Node Creation and Mapping
When an element is added, these steps are followed:
1. Create a clone of the element.
2. If there exists a node for the same identifier and Kind, call
handleDuplicate method on the existing node and return.
3. Create a new INode object wrapping the cloned element.
4. Store it in the nodes map using its identifier and Kind.
5. If there are pending edges in effectEdges or causeEdges, update the references between this node and the edges and remove
the entry from the corresponding map.
6. Process influence edges as described in Chapter 3.2.4.
3.2.2 Edge Creation and Mapping
When a relation is added, these steps are followed:
1. Create a clone of the relation.
2. If an edge with the same cause, effect, Kind, and optionally
identifier exists, call handleDuplicate on the edge and return.
3. Create a new IEdge object wrapping the cloned relation.
4. Extract the effect and cause identifiers from the relation using
ProvUtilities getEffect and getCause methods.
5. Extract the effect and cause Kind from the relation using ProvUtilities2 getEffectKind and getCauseKind methods.
6. Handle node references:
- If matching nodes exist, update the references.
- Else, store the edge in effectEdges or causeEdges or both.
7. Add the edge to the edges list.
The CpmDocument class contains a supporting method areAllRelationsMapped that checks whether every edge has references to corresponding cause and effect nodes.

It has been decided that an algorithm based on the presence of
traversal information specific attributes within element will be designed. Attributes allowed in traversal information are defined as
attributes belonging to the CPM namespace, with prefix "cpm" and IRI
"https://commonprovenancemodel.org/ns/cpm/", or the dct:hasPart attribute.
A known edge case comes from connector specializations. In CPM,
forward connector specializations are allowed in traversal information. However, domain-specific provenance is connected to traversal
information also by specializations. The algorithm then includes steps
to classify forward connector specializations as traversal information.
A consequence of this is that any specialization of a forward connector, containing no additional attributes, is classified as a traversal
information element.
The algorithm follows these steps:
1. If the element contains attributes that are not allowed in traversal information, classify it as a Domain-specific element.
2. If the element has a CPM prov:type and is not a specialization
of another element, classify it as a Traversal information element.
3. If the element is a specialization of more than one element,
classify it as Domain-specific element.
4. If the general element (i.e., the element from which the given
element is specialized) contains attributes that are not allowed
in traversal information, classify the specialized element as
Domain-specific element.
5. If the general element has a prov:type with value forwardConnector, and the specific element either has no prov:type
or has prov:type with value forwardConnector, classify it as
Traversal information element.
As this algorithm might be changed or replaced in the future, it has
been implemented using the strategy pattern by defining ITIStrategy
interface with boolean method belongsToTraversalInformation in
package model. The CpmDocument is initialized with the AttributeTIStrategy class from the strategy package, implementing the above
algorithm and can be replaced by a setter method.

3.4 CpmDocument component implementations
There are two different implementations of the interfaces defined in
Chapter 2.3.1 in the CPM library - divided and merged - and three
factories to instantiate them. The implementations differ in how they
handle addition of duplicate statements.
Duplicate statements are considered to be elements with the same
identifier and kind and relations with the same effect, cause and kind.
In the case of the Membership relation, if the identifier of the collection
and the list of members is the same, the relation is deemed a duplicate. The creation of identical membership and influence relations,
as described in Chapters 3.2.3 and 3.2.4 respectively, happens after
duplicates handling as described in Chapters 3.2.1 and 3.2.2.
In addition to instantiation of the nodes and edges, each factory
specifies a getComponentsTransformer that returns a transformer function from graph components to ProvToolBox statements, used when
converting back from CpmDocument to ProvToolBox Document, removing identical membership and influence relations generated during
their addition.
3.4.1 Merged Implementation
The merged implementations of nodes and edges, MergedNode and
MergedEdge, wrap exactly one Statement object inside. The handleDuplicate method utilizes a merging algorithm. Generally the lists of
attributes of the two elements are merged. Any duplicate attributes are
removed, keeping only one of each. The algorithm throws an exception
in these cases:
• For Activities: If there is a conflict in start or end times
• For Entities: If there is a conflict in the prov:value attribute
To utilize the Merged implementation, the CpmMergedFactory
needs to be passed as a constructor argument to the CpmDocument class.

3.4.2 Divided Implementation
The divided implementations of nodes and edges, DividedNode and

To keep the order, the CpmOrderedFactory utilizes an OrderManager
that keeps an identity-based map between statements and the order in
which they were added to the CpmDocument. If a statement is removed,
the order of the other statements does not change. If a new statement
is added, it is appended to the end. If a previously deleted statement
is added, it is appended to the end as well because a clone of the
statement, with a different identity, is created upon addition, as described in Chapters 3.2.1 and 3.2.2. When modifying the CpmDocument
using the methods defined in Appendix B.1, the order of statements
is not changed. When transforming back to ProvToolBox Document,
the getComponentsTransformer method utilizes the OrderManager to
sort the statements.

4.2 Comparison of template mechanisms
To compare the previous approach using the PROV template by Vojtěch Jurka [15] and the templating approach developed in this work
for the creation of traversal information, provenance documents containing the CPM bundles were created using both mechanisms.
To parse the XML data, Java objects representing the XML elements
have been created, and then the Jackson Dataformat XML library8 has
been used to deserialise the XML data into those objects. The process
only differed in the creation of traversal information, following these
general steps for each sample:
1. Create suffix for the sample by combining patient id and sample
id.
2. Create traversal information PROV document.
3. Add domain-specific provenance to the PROV document.
4.2.1 The PROV template mechanism
To create a document containing the traversal information using the
PROV template mechanism, first a PROV-N template document was
created for both the acquisition and the storage bundles. To populate
the document, for each sample these steps were followed:
1. Create a bindings file in-memory.
2. Combine bindings and template document to a PROV document.
The approach allows for separation of concerns; the logic is separated from the PROV-N template. This allows for more flexibility and
is easier to maintain by domain experts, as PROV-N is easier to read
for non-developers.
However, this approach is fragile, as mismatches between template
document and bindings can cause runtime issues and both files need to
be maintained and in sync. Furthermore, the responsibility of creating
8. https://github.com/FasterXML/jackson-dataformat-xml
39
4. Application of the Library to MMCI Data
both the template and bindings files for each topologically unique
traversal information part is put on a prospective user, raising the risk
of inconsistency between multiple systems and use cases.
The creation of the template document can be also done in-memory
using the functionalities of the ProvToolBox library. However, if this
more programmatic approach is chosen, in this specific case it is easier
to build the traversal information PROV document directly, bypassing
the templating and bindings mechanism altogether.
4.2.2 The CPM template mechanism
To create a document containing the traversal information using the
template mechanism defined in this work, these steps were followed
to create both acquisition and storage bundles for each sample:
1. Create traversal information object in-memory.
2. Map traversal information to a PROV document.
With this approach, only one file needs to be maintained. The
traversal information object serves as a replacement for the bindings
file, bypassing the need for a template file altogether. This approach
offers a layer of abstraction, as the structure maps directly to domain
concepts such as backward connector or main activity. Furthermore,
since the template will be maintained by the developers of the CPM,
it offloads the responsibility for creating the template for the traversal
information for each use case from a prospective user.
The cost of using this approach comes from less flexibility and expressivity, but this reduces the potential for errors. Another drawback
is that a prospective user must learn a new format, whereas the PROV
template mechanism is more commonly used.

5.2 Transformation of the provenance information
The provenance information have been provided in three different
formats: plain text, Ecological Metadata Language (EML)12, and JSONLD. For the purposes of this work, only the JSON-LD files are relevant.
These files are RDF-formatted and use the PROV-O [21] statement
definitions, however, they do not conform to the PROV JSON-LD
schema [8], and therefore require transformation. To address this,
an automated converter was implemented to perform the following
schema-specific modifications:
• Context Replacement: The original JSON-LD files use a custom
context. To apply the PROV JSON-LD context, the Titanium
JSON-LD13 library was used to perform these operations:
– The expand operation to remove the original context
– The compact operation to apply the PROV context
• Duplicate Relation Removal: In PROV-O [21], wasAssociatedWith and qualifiedAssociation provide different levels of
granularity. WasAssociatedWith represents only the binary relation itself and does not support additional attributes such as
prov:role or prov:label, whereas qualifiedAssociation allows
such additional information. However, in PROV JSON-LD this
distinction is not present. The same applies for wasAttributedTo
12. https://eml.ecoinformatics.org/
13. https://github.com/filip26/titanium-json-ld
43

and qualifiedAttribution. As a consequence, only qualifiedAttribution and qualifiedAssociation relations were kept.
• Fixing dcat:hadRole Usage: In the provided files, the dcat:
hadRole attribute is used to specify the role in all kinds of statements. However, according to the DCAT catalogue [22], this
attribute is to be used only with attribution relations. Therefore, dcat:hadRole is replaced with the role property for all
relevant statements, excluding attribution. This aligns with the
PROV JSON-LD specification where role is defined as a JSON
property corresponding to the prov:role attribute.
• Unnesting Elements: PROV JSON-LD does not support nested
structures in JSON objects. Nested relations and embedded
elements are therefore extracted and converted into stand-alone
JSON objects.
• Identifier Generation: Not all elements in the original files included identifiers. Since identifiers in elements are mandatory,
missing ones were generated by SHA-256 hashing of the corresponding nodes and multiple occurrences of identical nodes
were removed.
• Namespace Inference: Some identifiers were provided as IRIs
instead of qualified names. These IRIs were converted to qualified names, and the derived namespaces were added to the
context declaration.
• Miscellaneous Changes:
– Addition of the dcat namespace declaration
– Renaming of the schema:startTime and schema:endTime
attributes to prov:startTime and prov:endTime
– Replacement of Location kind with Entity, and Person/
Organisation kind with Agent
– Modification of the @type and prov:type properties to
PROV JSON-LD conventions
– Wrapping of primitive values according to PROV JSONLD conventions

5.4 Persistence of the documents
As a part of a master’s thesis by David Gajdoš [19], a Provenance
Storage service has been implemented for managing provenance documents. The service was built on top of the PROV Python package [13].
The bundles about EMBRC datasets described previously, each contained in a separate PROV document, were attempted to be stored in
this service in both the PROV-JSON [7] and PROV-XML [6] formats.
Even though both ProvToolBox and PROV Python package support
official PROV file formats, the documents generated by them are not
always compatible, analysed in detail by Matúš Formánek as part of
49
5. Application of the Library to EMBRC Data
his Bachelor’s thesis [23]. In addition to some of the issues described in
his work, several additional problems needed to be addressed, caused
by both the PROV Python package and the requirements of the service,
with their description following.
5.4.1 Incompatibilities between ProvToolBox and PROV Python
Blank namespace prefix
The original provenance information provided by EMBRC uses a blank
("_") namespace prefix without associating it with a namespace. While
ProvToolBox was able to work with this prefix by manual association
of the https://openprovenance.org/blank# namespace with the "_"
prefix, the Provenance Storage service was unable to deserialise PROVJSON documents with this prefix whatsoever in the PROV-JSON format. Documents in the PROV-XML format were deserialisable by the
service, but they weren’t serialisable back into PROV-JSON by the
service and as such this was not a proper fix. The solution was to
replace the "_" prefix with a "blank" prefix in all of its occurrences.
Inconsistent Bundle Identifier Handling in PROV-JSON
In PROV-JSON serialisation by ProvToolBox, the bundle identifier
is added as an @id property inside the bundle. This is not defined
in the official PROV-JSON schema14 and as such, the PROV Python
package was unable to deserialise such documents. The solution was
to manually remove this property after serialisation. An issue on the
ProvToolBox Github has been opened addressing this problem15
.
5.4.2 Requirements by the Provenance Storage service
The provenance storage defines constrains on the documents that
are stored in it, described in Chapter 3.2.1 of the Master’s thesis by
Gajdoš [19]. As noted in Chapter 5 of said thesis, it would be beneficial
for these constraints to be handled on behalf of the Provenance Storage
user, but as of writing of this thesis, this functionality has not been
implemented.

Storage and Meta namespace IRIs
When document is uploaded to the service, the namespace IRI of the
bundle’s identifier, and consequently the namespace IRI of the identifiers in referencedBundleId attributes in connectors, must match a
specific format to support the lookup of the bundle. The host in the
IRI must be the host of the service and an identifier of an organisation to which the document belongs must also be present. Furthermore, the namespace IRI of the meta bundle’s identifier, present in
the referencedMetaBundleId attribute in the main activity, is also required to match a specific format, with the host also corresponding to
the service’s host.
As both IRIs were unknown at the creation of the documents, the
associated namespace IRIs were required to be modified before the
documents were uploaded to the service.
Resolvability of referenced bundles
When documents are uploaded to the service, all referenced bundles
must be resolvable at the time of upload. Since the bundles for datasets
1 and 2 reference forthcoming bundles, they were required to be split
into two versions, V0 and V1. This is because the forthcoming bundles would be required to be already present in the service, which is
impossible, as they contain references to the preceding bundles.
Therefore, the V0 versions exclude forward connector specialisations referencing the forthcoming bundles. For consistency, the bundles for datasets 3 and 4, that do not reference forthcoming bundles,
are also treated as V0 versions. The correct upload sequence is to
submit all documents containing V0 bundles first, followed by the V1
versions for datasets 1 and 2, respectively.
The V0 versions use bundle identifiers with the suffix _V0, while
the V1 versions use suffix _V1. While the service does not enforce
a version naming convention, the chosen approach aligns with the
versioning shown in Figure 1.4.

When referencing another bundle in a connector, the referencedBundleHashValue and hashAlg attributes must be set to a hash of the

Conclusion
The goal of this thesis was to design and develop a reference implementation of the Common Provenance Model (CPM). The functionalities
of the library should have included creation of CPM bundles using
templates, representation of these bundles in-memory as traversable
graph, complete editability of these bundles, and various quality-oflife functionalities to support working with CPM.
All specified goals of this thesis have been accomplished. A library
has been created as an extension of an existing library, similarly to
how CPM is an extension of the PROV Data Model. Furthermore, the
functionalities of the library have been successfully demonstrated on
data provided by MMCI and EMBRC.
The creation of the reference implementation has already identified an issue in the latest specification of the CPM, namely the need to
differentiate specialisations of forward connectors between traversal
information and domain-specific provenance, as mentioned in Chapter 3.3. At the time of writing, a revision of the CPM has not yet been
finalized.
There are several areas in which the library developed in this
work could improve. First, a validator for CPM constraints in traversal
information could be useful. Second, currently there is no support for
versioning of the CPM specification. Since CPM bundles will be stored
long term, such a concept could prove essential, allowing seamless
future changes to the model. Open questions include: how to specify
the version of the CPM specification in the template, whether to persist
the version in the CPM bundle or its meta-provenance, and how to
identify relevant traversal information algorithm, as it might change
in the future, as touched upon in Chapter 3.3.
Introducing a new module to the library containing a validator for
the Provenance Storage service requirements described in Chapter 5.4
and a fully realised, properly tested converter would reduce the effort
needed to persist documents in the service. The work done in this area
in this thesis serves as a proof-of-concept, since a full realisation of
such a module fell outside the scope of this work.
53
Conclusion
In the long term, it would be beneficial to release the library in
the Maven repository, so it can be more easily integrated by other
developers and projects.

B.6 Requirements 6.6 and 6.7 - Retrieve preceding and
successive connectors
Precursors are defined as connectors from which the specified connector is derived, along with their own predecessors, and so on. Successive
connectors are defined as direct derivations of the specified connector,
along with their own derivations, and so on.
Two methods, using the Traversal Information Algorithm, are specified to retrieve these connectors, with outputs as follows:
• getPrecursors(QualifiedName id) /
getSuccessors(QualifiedName id)
– returns list of precursors or successors of the connector.
– returns empty list, if the connector does not have any precursors or successors.
– returns null, if a node with the specified id does not exist
or if it is not a connector.
Both methods use the same generalized private method to retrieve
the connectors using breadth-first search, as shown in Listing B.1.
private List<INode> getRelatedConnectors(QualifiedName id, Function<IEdge, INode>
extractNode, Function<INode, List<IEdge>> extractEdges) {
INode node = getNode(id, Kind.PROV_ENTITY);
if (!CpmUtilities.isConnector(node) || !tiStrategy.belongsToTraversalInformation(node)) {
return null;
}
List<INode> result = new ArrayList<>();
List<INode> toProcess = new ArrayList<>(extractEdges.apply(node).stream()
.map(extractNode)
.toList());
while (!toProcess.isEmpty()) {
INode current = toProcess.removeFirst();
result.add(current);
toProcess.addAll(extractEdges.apply(current).stream()
.filter(e −> Kind.PROV_DERIVATION.equals(e.getKind()))
.map(extractNode)
.filter(n −> node != n && !result.contains(n) &&
CpmUtilities.isConnector(n) &&
tiStrategy.belongsToTraversalInformation(node))
.toList());
}
return result;
}

B.7 Requirements 6.8 and 6.9 - Retrieve TI and DS as a
traversable sub-graph
Two methods, utilizing the same generalized private method, returning list of nodes satisfying the condition are defined:
• getTraversalInformationPart() /
getDomainSpecificPart()
– returns list of deep-cloned nodes of the given part, containing references to edges only between other nodes within
the same part.
To create a traversable sub-graph, each node and edge needs to be
essentially deep-cloned. The whole process, shown in Listing B.2 is
described as follows.
To minimize redundant computations, first a map of original nodes
to cloned nodes is created, filtered by the Traversal Information Algorithm. Then all edges that are between nodes present in the map are
retrieved. Edges that are not fully mapped, e.g. they are missing cause
or effect node, are ignored, as the filter that checks their presence in
clonedNodeMap will simply return false. Lastly for each edge, a clone
is created, and the cloned cause and effect nodes are set. Cloning the
node and edge, using CpmFactory (cF) methods removes references
to other nodes and edges. This way, no connection to the other part of
the bundle is present in the returned sub-graph.

private List<INode> getFilteredNodes(Predicate<INode> nodeFilter) {
Map<INode, INode> clonedNodeMap = getNodes().stream()
.filter(nodeFilter)
.collect(Collectors.toMap(node −> node, cF::newNode));
List<IEdge> edges = clonedNodeMap.keySet().stream()
.flatMap(n −> n.getCauseEdges().stream()
.filter(e −> clonedNodeMap.containsKey(e.getEffect())))
.toList();
for (IEdge edge : edges) {
IEdge clonedEdge = cF.newEdge(edge);
clonedEdge.setCause(clonedNodeMap.get(edge.getCause()));
clonedEdge.getCause().getCauseEdges().add(clonedEdge);
clonedEdge.setEffect(clonedNodeMap.get(edge.getEffect()));
clonedEdge.getEffect().getEffectEdges().add(clonedEdge);
}
return clonedNodeMap.values().stream().toList();
}
public List<INode> getTraversalInformationPart() {

public List<INode> getDomainSpecificPart() {
