\addcontentsline{toc}{chapter}{Introduction}

In the era of data-driven research and complex computational workflows, tracking
the origin, transformation, and lineage of data has become essential for ensuring
reproducibility, transparency, and trustworthiness of scientific results. This
challenge is addressed by the field of data provenance, which provides systematic
approaches to recording and querying the history of data artifacts.

The importance of provenance has grown dramatically in recent years. Scientific
publications increasingly require data availability statements and reproducibility
documentation. Regulatory frameworks in healthcare, finance, and government mandate
audit trails for data processing. Machine learning applications need to track
training data lineage to ensure model fairness and detect data poisoning. In all
these contexts, understanding where data came from, how it was transformed, and
who was responsible for each step has become not just useful but essential.

The World Wide Web Consortium (W3C) standardized the PROV Data Model (PROV-DM)~\cite{provdm}
as a generic foundation for representing provenance information. This standard
defines core concepts—entities, activities, and agents—along with relationships
between them such as generation, usage, derivation, and attribution. PROV-DM
has achieved broad adoption across domains~\cite{moreau2013prov}, with implementations available in
multiple programming languages and integration with semantic web technologies.

However, the generality of PROV-DM, while enabling broad applicability, often
proves insufficient for domain-specific applications that require structured
provenance chains with well-defined traversal patterns and semantic relationships.
Applications that need to traverse provenance graphs across organizational
boundaries, aggregate provenance from multiple sources, or distinguish between
infrastructure-level and domain-level provenance find themselves implementing
ad-hoc extensions that may not interoperate with other systems.

The Common Provenance Model (CPM)~\cite{wittner2022, wittner2023cpm}
extends PROV-DM to address these limitations by
introducing a structured approach to provenance representation. CPM defines
specific component types—Main Activities, Forward Connectors, Backward Connectors,
and Identifier Entities—that enable systematic traversal of provenance chains
across distributed systems while maintaining clear separation between traversal
infrastructure and domain-specific content. This separation is fundamental to
CPM's design: it allows provenance consumers to navigate complex, distributed
provenance chains by following only the traversal infrastructure, while still
having access to rich domain-specific content when needed.

A reference implementation of CPM exists in Java, providing mature and well-tested
support for CPM concepts. However, Python has become the dominant language for
data science, scientific computing, and machine learning—precisely the domains
where provenance tracking is most critical. Python developers currently lack
native CPM support, forcing them to either implement CPM concepts from scratch,
integrate with Java through complex inter-process communication, or forego
CPM's benefits entirely.

This thesis addresses this gap by presenting the design, implementation, and
evaluation of a complete Python library for the Common Provenance Model. The
library provides full CPM functionality including document management, component
classification, template-based workflows, comprehensive validation, and efficient
graph operations. The implementation maintains compatibility with the Java
reference implementation through verified template serialization and consistent
classification behavior, enabling seamless provenance exchange between Python
and Java systems.

The thesis is organized as follows. Chapter~\ref{chap:theoretical-background}
provides theoretical background on data provenance, the W3C PROV standard, and
the Common Provenance Model specification. Chapter~\ref{chap:analysis-requirements}
analyzes existing implementations and derives requirements for the Python library.
Chapter~\ref{chap:design} presents the architectural design and key design
decisions. Chapter~\ref{chap:implementation} details the implementation of
major components. Chapter~\ref{chap:usage-examples} demonstrates practical usage
through examples. Chapter~\ref{chap:evaluation-testing} evaluates the implementation
through testing and performance analysis. Chapter~\ref{chap:conclusion} concludes
with a summary of achievements and directions for future work.

\chapter{Theoretical Background}
\label{chap:theoretical-background}

This chapter provides the theoretical foundation necessary for understanding
the Common Provenance Model implementation. We begin with an introduction to
data provenance concepts, followed by a detailed examination of the W3C PROV
standard, and conclude with the Common Provenance Model specification.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Data Provenance}
\label{sec:data-provenance}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

Data provenance, derived from the French word \textit{provenir} meaning "to come from,"
refers to the documented history of data including its origins, transformations,
and the processes that led to its current state~\cite{simmhan2005}. In the context of scientific
computing and data management, provenance serves as a comprehensive audit trail
that enables researchers and practitioners to understand, reproduce, and validate
computational results.

The concept of provenance has deep roots in archival science and museum curation,
where documenting the ownership history of artifacts has been essential practice
for centuries. In the digital age, this concept has been adapted to track the
lineage of data and computational results. Early work on database provenance in
the 1990s established foundational concepts~\cite{buneman2001},
while the explosion of data-intensive
science in the 2000s brought provenance to the forefront of computer science
research~\cite{freire2008}.

The importance of data provenance has grown significantly with the increasing
complexity of data-driven workflows. Modern scientific investigations often
involve multiple data sources, complex transformation pipelines, and distributed
computing environments. A single scientific result may depend on hundreds of
intermediate computations, each potentially drawing from different data sources
and using different processing methods. Without proper provenance tracking, it
becomes nearly impossible to reproduce results, debug errors, assess data quality,
comply with regulations, or enable effective collaboration.

Consider a typical machine learning pipeline: raw data is collected from multiple
sources, cleaned and preprocessed, features are extracted and selected, models
are trained with specific hyperparameters, and predictions are generated. Each
step involves choices that affect the final result. Provenance tracking captures
all these choices, enabling researchers to understand why a model behaves as it
does and to reproduce the entire pipeline when needed.

Provenance information can be categorized along several dimensions~\cite{freire2008}. 
\textbf{Prospective provenance} describes the computational procedure or workflow
specification that will be executed, capturing the intended sequence of operations.
This includes workflow definitions, scripts, and configuration files that specify
what should happen. \textbf{Retrospective provenance} records what actually happened
during execution, including actual data values, timestamps, execution environment
details, and any deviations from the prospective plan. Both types are valuable:
prospective provenance enables reproduction, while retrospective provenance
enables debugging and auditing.

Provenance can also be classified by granularity. \textbf{Fine-grained provenance}
tracks individual data items at a detailed level—for example, recording which
input tuples contributed to each output tuple in a database query. This level of
detail supports precise debugging and attribution but can generate enormous amounts
of metadata. \textbf{Coarse-grained provenance} operates at higher levels of
abstraction, tracking relationships between entire datasets or workflow steps.
This approach is more practical for large-scale systems but may not provide
enough detail for some use cases.

The provenance research community identifies three fundamental questions that
provenance systems should answer~\cite{cheney2009, buneman2001}.
\textbf{Why-provenance} asks which input data
items contributed to producing a particular output—this is essential for debugging
unexpected results and understanding data dependencies. \textbf{How-provenance}
asks what process or derivation steps led to the output—this captures the
computational logic and transformation sequence. \textbf{Where-provenance} asks
from which exact location in source data were output values copied or derived—
this is important for data integration scenarios where values may come from
multiple sources.

Beyond these core questions, provenance systems increasingly need to address
questions of responsibility (who performed or authorized each action), timing
(when did events occur and in what order), and context (what was the computational
environment, what parameters were used). Modern provenance frameworks aim to
capture all these dimensions in an integrated model.



%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{W3C PROV Standard}
\label{sec:w3c-prov}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

The W3C Provenance Working Group published a family of specifications in 2013~\cite{provoverview}
that define a standard model for representing and exchanging provenance
information on the Web. This standardization effort was motivated by the
proliferation of incompatible provenance
systems~\cite{moreau2011opm} and the need for interoperability
across organizational and system boundaries. The resulting PROV family of
standards has achieved widespread adoption and forms the foundation for most
modern provenance systems.

The PROV family consists of several interrelated documents, each addressing
different aspects of provenance representation and exchange. \textbf{PROV-DM}
(the Data Model)~\cite{provdm} defines the conceptual data model, providing the core
abstractions that all other specifications build upon. \textbf{PROV-O} (the
Ontology)~\cite{provo} provides an OWL2 ontology for representing PROV in RDF, enabling
integration with semantic web technologies. \textbf{PROV-N} (the Notation)~\cite{provn}
offers a human-readable notation for PROV expressions, useful for documentation
and debugging. \textbf{PROV-CONSTRAINTS}~\cite{provconstraints} defines validity constraints that
PROV documents should satisfy, enabling automated validation. \textbf{PROV-LINKS}~\cite{provlinks}
defines mechanisms for connecting provenance bundles and documents, supporting
distributed provenance scenarios.

\subsection{Core Types and Relations}

PROV-DM defines three core types that form the foundation of all provenance
representations. An \textbf{Entity} is a physical, digital, conceptual, or
other kind of thing with some fixed aspects. In data provenance, entities
typically represent data items, files, documents, or other artifacts that
can be created, used, and transformed. Entities are characterized by their
immutability—once an entity exists, its essential characteristics are fixed.
If a data file is modified, the result is conceptually a new entity derived
from the original.

An \textbf{Activity} is something that occurs over a period of time and acts
upon entities. Activities represent the dynamic aspects of provenance—the
processes, computations, or actions that create, modify, or use data. Each
activity has a defined start and end time (though these may be unknown in
practice) and may use entities as inputs and generate entities as outputs.
Activities can be nested or composed, with complex workflows represented as
hierarchies of activities.

An \textbf{Agent} bears responsibility for an activity or entity. Agents can
be people, organizations, or software systems. The agent concept enables
attribution—we can record not just what happened but who (or what) was
responsible for making it happen. Multiple agents may be associated with a
single activity with different roles, and agents can delegate authority to
other agents.

\begin{lstlisting}
entity(ex:dataset1, [prov:type='prov:Collection'])
activity(ex:analysis1, 2024-01-15T10:00:00, 2024-01-15T11:30:00)
agent(ex:researcher1, [prov:type='prov:Person'])
\end{lstlisting}

PROV-DM defines several relations connecting these types. \textbf{Generation}
(\texttt{wasGeneratedBy}) represents production of an entity by an activity—
the activity created the entity as an output. \textbf{Usage} (\texttt{used})
represents utilization of an entity by an activity—the activity consumed or
referenced the entity as an input. \textbf{Derivation} (\texttt{wasDerivedFrom})
represents transformation of one entity into another, capturing the dependency
between data items. \textbf{Attribution} (\texttt{wasAttributedTo}) ascribes
an entity to an agent, recording responsibility for the entity's existence.
\textbf{Association} (\texttt{wasAssociatedWith}) assigns responsibility for
an activity to an agent. \textbf{Delegation} (\texttt{actedOnBehalfOf})
represents transfer of authority between agents.

These relations can carry additional information. Generation and usage can
include timestamps and roles. Derivation can specify the activity that
performed the transformation. Association can include a plan that the agent
was following. This richness enables detailed provenance capture while
maintaining a clean conceptual model.

\subsection{PROV-N Notation}

PROV-N provides a human-readable syntax for PROV expressions that is more
accessible than RDF or XML representations while being formally defined and
machine-parseable:

\begin{lstlisting}
document
  prefix ex <http://example.org/>
  
  entity(ex:article, [prov:type='ex:JournalArticle',
                      prov:label='Research Paper'])
  activity(ex:compose, 2024-01-10T09:00:00, 2024-01-15T17:00:00,
           [prov:type='ex:Writing'])
  agent(ex:derek, [prov:type='prov:Person',
                   foaf:name='Derek Author'])
  
  wasGeneratedBy(ex:article, ex:compose, 2024-01-15T17:00:00)
  wasAssociatedWith(ex:compose, ex:derek, -)
  wasAttributedTo(ex:article, ex:derek)
endDocument
\end{lstlisting}

The notation uses prefixes for namespace management, supports optional
parameters indicated by dashes, and allows arbitrary key-value attributes
in square brackets. This readability makes PROV-N valuable for documentation,
debugging, and communication between provenance system developers.

\subsection{Constraints and Bundles}

PROV-CONSTRAINTS defines validity rules ensuring logical consistency of
provenance documents. These include typing constraints (relations must connect
appropriate types—for example, generation connects an entity to an activity),
temporal constraints (entities cannot be used before generation, activities
have consistent time bounds), uniqueness constraints (certain properties have
unique values), and ordering constraints (events have consistent temporal
ordering).

Bundles are named sets of provenance descriptions that can themselves have
provenance. This mechanism enables grouping of related statements into logical
units, meta-provenance (provenance about provenance—who recorded the provenance
and when), and modularizing large provenance graphs for distributed scenarios.
A bundle appears as an entity, allowing provenance statements about the bundle
itself, such as who created it and when.

The bundle mechanism is essential for distributed provenance because it allows
different systems to contribute provenance about their portion of a workflow
while maintaining clear boundaries and enabling provenance about the provenance
contributions themselves.

\section{Common Provenance Model (CPM)}
\label{sec:cpm}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

The Common Provenance Model (CPM) is a structured provenance model that extends
W3C PROV-DM to support systematic representation and traversal of provenance
chains across distributed systems~\cite{wittner2022, wittner2023cpm}.
While PROV-DM provides a flexible foundation
for provenance representation, its generality can be problematic for applications
requiring structured traversal patterns, clear component separation, cross-system
interoperability, and scalable provenance aggregation. CPM addresses these
challenges by defining specific structural patterns and component types that
enable uniform handling of provenance across diverse systems.

The motivation for CPM arises from practical challenges in distributed provenance
scenarios. Consider a scientific workflow that spans multiple institutions: data
is collected at one site, processed at a computing center, analyzed by researchers
at a university, and published through a repository. Each site may use different
provenance systems, and the complete provenance chain must somehow be assembled
from these disparate sources. Without a common structure, traversing such
distributed provenance becomes extremely difficult—each system may represent
similar concepts differently, use incompatible identifiers, and structure its
provenance in unique ways.

CPM solves this problem by defining a common structure that all participating
systems can adopt. By agreeing on how to represent key structural elements—
the boundaries between systems, the connection points for traversal, and the
separation between infrastructure and domain content—systems can participate
in distributed provenance scenarios without tight coupling.

\subsection{TI/DS Separation}

A fundamental concept in CPM is the separation between \textbf{Traversal
Information (TI)} and \textbf{Domain Specific (DS)} content. Traversal
information comprises the infrastructure components that enable navigation
between provenance chains—the structural elements that a provenance consumer
needs to follow when tracing lineage across systems. Domain-specific content
comprises the actual domain-relevant provenance data—the detailed information
about what happened within a particular processing step.

This separation serves several purposes. First, it enables efficient traversal:
a consumer interested only in understanding the high-level lineage of a data
product can follow TI components without processing potentially enormous amounts
of domain-specific detail. Second, it promotes interoperability: the TI layer
can be standardized across systems while the DS layer accommodates domain-specific
requirements. Third, it supports layered querying: consumers can first navigate
using TI to locate relevant portions of a provenance graph, then drill into DS
content for detailed analysis.

The classification of components as TI or DS follows defined rules based on
type annotations and structural relationships. Components explicitly typed as
CPM structural elements belong to TI. Entities directly connected to TI activities
through structural relationships (generation or usage by connectors) belong to
TI. Agents associated with TI activities belong to TI. All other components
belong to DS.

\subsection{CPM Component Types}

CPM defines several specific component types that provide the structural
framework for provenance chains.

The \textbf{Main Activity} is the central component of a CPM provenance chain,
representing the primary processing step. Exactly one Main Activity exists per
CPM bundle, serving as the anchor point that connects all other CPM components.
The Main Activity typically represents the core processing—the actual work
performed by the system contributing this provenance. It is connected to
connectors that handle input acquisition and output distribution, and to agents
responsible for the processing.

\textbf{Forward Connectors} precede the Main Activity in the provenance flow,
representing operations that prepare or provide inputs. A Forward Connector
generates entities that are consumed by the Main Activity. This pattern
enables linking to preceding provenance chains: the entity generated by a
Forward Connector may be the same entity (or derived from an entity) that
appears as an output in another system's provenance, creating the link between
chains. Forward Connectors thus serve as the "input ports" of a CPM bundle.

\textbf{Backward Connectors} follow the Main Activity in the provenance flow,
representing operations that process or distribute outputs. A Backward
Connector uses entities generated by the Main Activity. This pattern enables
linking to subsequent provenance chains: the entity consumed by a Backward
Connector may appear as an input in another system's provenance. Backward
Connectors serve as the "output ports" of a CPM bundle.

\textbf{Identifier Entities} carry identification information for linking
provenance chains across systems. These entities hold persistent identifiers
such as DOIs, URIs, or other cross-reference information that enables consumers
to discover related provenance chains. Identifier Entities are associated with
the Main Activity and support the resolution of provenance across system
boundaries.

The following example illustrates a complete CPM bundle structure:

\begin{lstlisting}
bundle ex:provenanceBundle
  // Main Activity - the core processing
  activity(ex:main, [prov:type='cpm:MainActivity',
                     prov:label='Data Analysis'])
  
  // Forward Connector - handles input acquisition
  activity(ex:fc1, [prov:type='cpm:ForwardConnector',
                    prov:label='Data Loader'])
  
  // Backward Connector - handles output distribution
  activity(ex:bc1, [prov:type='cpm:BackwardConnector',
                    prov:label='Result Publisher'])
  
  // Entities connecting the activities
  entity(ex:input1, [prov:label='Input Dataset'])
  entity(ex:output1, [prov:label='Analysis Results'])
  
  // Structural relationships
  wasGeneratedBy(ex:input1, ex:fc1)  // FC generates input
  used(ex:main, ex:input1)            // Main uses input
  wasGeneratedBy(ex:output1, ex:main) // Main generates output
  used(ex:bc1, ex:output1)            // BC uses output
endBundle
\end{lstlisting}

\subsection{Validation Rules}

CPM documents must satisfy validation rules that ensure structural correctness
and semantic consistency. These rules are categorized by type.

Structural rules verify the component composition of a CPM document. The
fundamental rule requires exactly one Main Activity per bundle—zero Main
Activities indicates an incomplete CPM document, while multiple Main Activities
suggests that the bundle should be split. Connector rules verify that Forward
Connectors generate entities used by the Main Activity (establishing proper
input flow) and that Backward Connectors use entities generated by the Main
Activity (establishing proper output flow). Type annotation rules verify that
all CPM components have correct \texttt{prov:type} attributes with appropriate
CPM namespace URIs.

Connectivity rules verify the graph structure of a CPM document. The primary
connectivity rule requires that all CPM components are reachable from the
Main Activity through some path—orphaned components indicate incomplete
provenance that may cause traversal problems. Relationship direction rules
verify that generation relations point from activities to entities and usage
relations point from activities to entities.

Consistency rules verify logical coherence. Temporal rules check that activities'
start times precede their end times and that entities are not used before they
are generated. Type consistency rules verify that relations connect appropriate
element types. Identifier uniqueness rules verify that no two elements share
the same identifier within a bundle.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Existing Implementations}
\label{sec:existing-implementations}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

Before developing a new implementation, we must examine existing solutions to
understand the current landscape, learn from prior design decisions, and
identify gaps that motivate this work. This analysis focuses on the Java
reference implementation and the Python PROV library.

\subsection{Java Reference Implementation}

The primary reference implementation of the Common Provenance Model was
developed in Java during prior research at Masaryk University. This
implementation serves as the authoritative source for CPM behavior—when
questions arise about how a CPM feature should work, the Java implementation
provides the definitive answer.

The Java implementation is built on ProvToolbox~\cite{provtoolbox}, a mature Java library for
W3C PROV-DM that provides comprehensive support for PROV types, relations,
and serialization formats. This foundation enabled the CPM implementation
to focus on CPM-specific functionality rather than reimplementing base PROV
capabilities.

The implementation provides complete CPM functionality across several areas.
For document management, it supports creating CPM documents, wrapping existing
PROV documents, adding and removing components, and querying document contents.
For component support, it implements all CPM component types including Main
Activity, Forward Connector, Backward Connector, and Identifier Entity with
proper type annotations and relationship management. The TI/DS separation
algorithm correctly classifies components as traversal information or
domain-specific content based on CPM rules.

The template system is particularly noteworthy. Templates enable declarative
specification of CPM documents through a structured format that captures all
necessary information—component identifiers, labels, relationships, and
namespace bindings. Templates can be serialized to JSON for storage and
exchange, and deserialized to reconstruct documents. This mechanism supports
configuration-driven provenance capture and enables provenance exchange between
systems through a well-defined JSON format.

Validation in the Java implementation checks structural rules (Main Activity
presence, connector relationships), connectivity rules (component reachability),
and consistency rules (temporal ordering, type correctness). Validation results
are categorized by severity and include detailed messages identifying
problematic components.

The implementation uses factory patterns extensively, with separate factories
for different component configurations. Documents are treated as effectively
immutable—modification operations return new documents rather than changing
existing ones. This design choice simplifies reasoning about document state
but may have performance implications for workflows involving many modifications.
A comprehensive JUnit test suite with hundreds of test cases ensures correctness
and prevents regressions.

\subsection{PROV Python Library}
\label{subsec:prov-python-analysis}

The \texttt{prov} library~\cite{provlib} provides a comprehensive Python implementation of
PROV-DM, serving as a potential foundation for any CPM implementation in Python.
Developed and maintained as an open-source project, it has become the de facto
standard for PROV work in Python with wide adoption in research and industry.

The library implements the complete PROV-DM type system including entities,
activities, agents, and all standard relations. It provides multiple serialization
formats: PROV-JSON for data exchange, PROV-N for human-readable notation, RDF
for semantic web integration through multiple serializations (Turtle, RDF/XML,
N-Triples), and PROV-XML for XML-based exchange. The library handles namespace
management with support for standard PROV namespaces and custom definitions.
Bundle support enables organization of provenance statements into logical units.

The following example demonstrates the library's capabilities:

\begin{lstlisting}
from prov.model import ProvDocument

# Create a new provenance document
doc = ProvDocument()

# Add namespace bindings
doc.add_namespace('ex', 'http://example.org/')
doc.add_namespace('custom', 'http://example.org/custom/')

# Create entities with attributes
e1 = doc.entity('ex:data', {
    'prov:label': 'Input Dataset',
    'prov:type': 'custom:Dataset',
    'custom:size': '10GB'
})

# Create activities with timestamps
a1 = doc.activity('ex:process', 
    startTime='2024-01-15T10:00:00',
    endTime='2024-01-15T11:30:00',
    other_attributes={'prov:label': 'Data Processing'})

# Create agents
ag1 = doc.agent('ex:system', {
    'prov:type': 'prov:SoftwareAgent',
    'prov:label': 'Processing System v2.0'
})

# Create relationships
doc.wasGeneratedBy(e1, a1)
doc.wasAssociatedWith(a1, ag1)

# Serialize to different formats
json_str = doc.serialize(format='json')
provn_str = doc.serialize(format='provn')
\end{lstlisting}

The library provides rich functionality for document manipulation including
element creation with arbitrary attributes, efficient element lookup by
identifier, iteration over elements by type, and document merging for
combining provenance from multiple sources.

However, the library implements only the base PROV-DM specification. It has no
knowledge of CPM concepts—Main Activities, Connectors, and Identifier Entities
are not recognized as distinct component types. Users can construct these
components manually by creating PROV elements with appropriate type annotations,
but the library provides no assistance or validation for CPM structure.

There is no mechanism for distinguishing traversal information from domain-specific
content. Users who need this separation must implement the classification logic
themselves, risking inconsistency with the reference implementation. There are
no CPM-specific validation rules—the library can check basic PROV constraints
but cannot verify CPM structural requirements like Main Activity uniqueness or
proper connector relationships.

The library has no template mechanism for declarative document specification.
Users must construct documents imperatively, which can be verbose and error-prone
for complex structures. There are no CPM-aware graph operations—while basic
traversal is possible through the library's API, there is no optimized support
for operations like subgraph extraction based on TI/DS classification.

\subsection{Gap Analysis}

The analysis of existing implementations reveals a significant gap in the
Python ecosystem. Java developers have access to a mature, well-tested CPM
library that provides all needed functionality with good integration into
Java development workflows. The library handles complex concerns like
template serialization, component classification, and validation, allowing
developers to focus on their domain-specific provenance needs.

Python developers, in contrast, must choose between unsatisfactory options.
They can use the base \texttt{prov} library~\cite{provlib} and implement all CPM logic from
scratch, but this requires significant effort and risks behavioral differences
from the reference implementation that could cause interoperability problems.
They can integrate with the Java implementation through inter-process
communication, but this adds substantial complexity, performance overhead,
and deployment challenges. Or they can avoid CPM entirely, using only base
PROV capabilities, but this foregoes the benefits of structured provenance
chains and TI/DS separation.

This gap is particularly problematic given Python's dominant position in
data science, scientific computing, and machine learning—precisely the domains
where provenance tracking is most valuable. Researchers using Python for
computational experiments, data engineers building Python-based pipelines,
and developers creating Python applications all lack native CPM support.

This gap motivates the development of a native Python CPM library that provides
equivalent functionality to the Java reference implementation while following
Python conventions and integrating smoothly with the Python ecosystem. Such a
library would enable Python developers to participate fully in CPM-based
provenance scenarios and interoperate seamlessly with Java systems.

\section{Target Users and Use Cases}
\label{sec:use-cases}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

Understanding the target users and their use cases is essential for deriving
appropriate requirements. The Python CPM library serves several distinct user
groups with different needs and expectations.

\textbf{Research scientists} work with computational experiments and data
analysis pipelines where provenance is essential for reproducibility and
scientific integrity. They typically work in Jupyter notebooks and use Python
scientific libraries like NumPy, Pandas, and scikit-learn. They need simple,
intuitive APIs that don't require deep understanding of provenance theory.
Clear documentation with examples is essential—they want to see how to apply
the library to scenarios similar to their own. They may work with relatively
small provenance graphs (tens to hundreds of nodes) but need reliable
serialization for sharing and archiving.

\textbf{Data engineers} build production data pipelines where provenance
supports auditing, debugging, and compliance requirements. They need
programmatic APIs that integrate well with pipeline frameworks and can be
called from automated processes. Performance matters—operations should
complete quickly even for larger graphs, and memory usage should be
predictable. Serialization format compatibility is critical for integrating
with existing systems, particularly those using Java. They expect robust
error handling and clear exception hierarchies.

\textbf{Software developers} building provenance-aware applications expect
a polished library experience. They want type hints for IDE support and
static type checking. They expect consistent naming conventions following
Python standards (snake\_case for functions, PascalCase for classes). They
need comprehensive documentation including API reference, tutorials, and
examples. High test coverage gives confidence in the library's reliability.
They may need to extend the library and expect a clean, well-organized
codebase.

\textbf{Interoperability use cases} are particularly important given the
existence of the Java reference implementation. Organizations using both
Python and Java systems need seamless provenance exchange. Python components
in a pipeline must produce provenance that Java components can consume, and
vice versa. This requires compatible template serialization (identical JSON
format), consistent component classification (same TI/DS decisions), and
equivalent validation results (same errors and warnings for the same documents).

Common workflow patterns include capturing provenance during Python script
execution, serializing to JSON for storage or exchange, deserializing in
Java for aggregation or analysis, and vice versa. The library must support
these patterns without introducing subtle incompatibilities that would cause
integration failures.

\section{Functional Requirements}
\label{sec:functional-requirements}

The library must provide complete \textbf{document management} including
creating CPM documents, adding/retrieving/updating/removing nodes and edges,
and converting between CPM and PROV representations.

Full \textbf{CPM component support} is required: Main Activities with unique
identifiers and optional timestamps, Forward and Backward Connectors with

correct type annotations for all components.

The library must implement \textbf{TI/DS separation} matching reference
implementation behavior, with methods to extract traversal information or
domain-specific content and create filtered subgraphs.

A \textbf{template system} is required for declarative document specification

supporting roundtrip conversion without information loss.

\textbf{Graph operations} must include traversal, path finding, predecessor
and successor queries, and subgraph extraction, preferably through integration
with established graph libraries.

Comprehensive \textbf{validation} must check structural rules (exactly one
Main Activity, proper connector relationships), connectivity rules (all
components connected), and consistency rules (temporal, type, identifier
uniqueness), with results categorized by severity.

Multiple \textbf{serialization formats} must be supported: JSON through
templates, PROV-N, PROV-JSON, and RDF.

Beyond functional capabilities, the library must satisfy quality requirements

\textbf{Maintainability} requires logical module organization with clear
responsibilities, type hints for all public APIs, code passing standard
linting tools, and minimal well-justified dependencies.

\textbf{Extensibility} means new component types, serialization formats,
validation rules, and graph operations can be added without modifying
existing code.

\chapter{Design}
\label{chap:design}

This chapter presents the design decisions for the Python CPM library. We describe
the overall architecture, design patterns employed, and the rationale behind key
design choices that address the requirements established in the previous chapter.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Architecture Overview}
\label{sec:architecture-overview}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

The Python CPM implementation follows a layered architecture that separates
concerns while providing a cohesive API to users. This architectural approach
enables clean separation of responsibilities, facilitates testing at multiple
levels, and allows each layer to evolve independently while maintaining
well-defined interfaces.

At the foundation lies the \textbf{PROV layer}, provided by the external
\texttt{prov} library~\cite{provlib}. This layer handles all W3C PROV-DM concerns including
the core type system (entities, activities, agents), relations (generation,
usage, derivation, attribution, association, delegation), namespace management,
bundle support, and serialization to standard formats. The CPM implementation
treats this layer as stable infrastructure, depending on its public API without
modifying or extending its internals.

Above the PROV layer sits the \textbf{graph abstraction layer}. This layer
wraps PROV documents in graph-oriented abstractions that hide PROV-specific
details and provide optimized operations. The key insight is that provenance
documents are fundamentally graphs—nodes connected by edges—and many important
operations are naturally expressed as graph algorithms. Rather than implementing
graph algorithms from scratch, this layer integrates with NetworkX~\cite{networkx}, a mature
Python library for graph analysis. The dual representation—maintaining both
the PROV document for compliance and a NetworkX graph for algorithms—enables
the best of both worlds.

At the top sits the \textbf{CPM layer}, implementing CPM-specific functionality.
This layer adds CPM concepts including component classification (Main Activity,
Connectors, Identifier Entities), the TI/DS separation algorithm, templates
for declarative document specification, and CPM validation rules. The CPM
layer uses the graph abstraction layer for traversal and analysis while
maintaining CPM semantics.

The source code organization reflects this architecture. The \texttt{src/cpm}
package contains CPM-specific implementation: the document model in
\texttt{model/}, templates in \texttt{template.py}, validation in
\texttt{validation.py}, and the builder in \texttt{builder.py}. The
\texttt{src/graph} package provides graph abstractions including wrappers,
nodes, edges, and factories. The \texttt{src/adapters} package contains
utilities for PROV integration. The \texttt{src/utils} package provides
shared utility functions. This organization makes it easy to locate
functionality and understand dependencies—the CPM package depends on the
graph package, which depends on adapters and utilities, creating a clear
dependency hierarchy without cycles.


%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{CpmDocument Design}
\label{sec:cpmdocument-design}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

The \texttt{CpmDocument} class serves as the primary entry point for users,
providing a comprehensive API for CPM document manipulation. The design
balances extensive functionality with ease of use and maintainability through
careful application of object-oriented design principles.

\subsection{Mixin Architecture}

Rather than implementing all functionality in a single monolithic class,
\texttt{CpmDocument} is composed from several focused mixins. Each mixin
provides a specific category of operations, can be developed and tested
independently, and keeps the overall class organized despite its extensive API.

The \texttt{CoreMixin} provides fundamental CRUD (Create, Read, Update, Delete)
operations for nodes and edges. For nodes, this includes \texttt{add\_node}
for creating entities, activities, or agents with specified identifiers and
attributes; \texttt{get\_node} for retrieving a single node by identifier;
\texttt{get\_nodes} for querying nodes by type or attribute predicates;
\texttt{update\_node} for modifying node attributes; and \texttt{remove\_node}
for deletion with configurable handling of connected edges. Parallel methods
exist for edge operations.

The \texttt{IOMixin} handles all serialization and format conversion. It
provides \texttt{to\_json} for JSON export, \texttt{to\_prov\_document} for
extracting the underlying PROV representation, \texttt{to\_provn} for
human-readable PROV-N output, and methods for template conversion. The
mixin encapsulates format-specific logic, making it straightforward to add
new formats without affecting other functionality.

The \texttt{AnalysisMixin} provides analytical operations that help users
understand and verify their documents. The \texttt{get\_statistics} method
returns metrics including node counts by type, edge counts by relation type,
and component classification summaries. The \texttt{validate} method performs
comprehensive CPM compliance checking and returns categorized results. Methods
for accessing TI and DS components enable selective processing.

The \texttt{TraversalMixin} provides graph navigation capabilities:
\texttt{get\_predecessors} and \texttt{get\_successors} for immediate
neighbors, \texttt{find\_paths} for discovering all paths between two
nodes, \texttt{traverse\_from} for general graph exploration with
customizable filtering, and \texttt{create\_subgraph} for extracting
portions of the document based on predicates.

\subsection{Document State and Caching}

A \texttt{CpmDocument} wraps an underlying PROV document while maintaining
additional state for CPM functionality. The class stores a reference to the
PROV document, namespace bindings for identifier resolution, and cached
analysis results for performance.

The design ensures that modifications through the CPM API are immediately
reflected in the underlying PROV document. When users call methods like
\texttt{add\_node} or \texttt{add\_edge}, the changes are applied to the
PROV document synchronously. Cached values such as component classifications
and statistics are invalidated on modification and recomputed on next access.
This approach balances performance with correctness—frequently accessed
values are cached to avoid redundant computation, but modifications always
produce correct results without stale cache issues.

Factory methods supplement the standard constructor. The \texttt{from\_template}
class method creates a document from a template specification, handling the
conversion internally. The \texttt{from\_prov\_file} method loads a PROV
document from a file in any supported format and wraps it. These factory
methods provide clean APIs for common creation patterns.


%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Graph Abstraction Layer}
\label{sec:graph-abstraction}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

The graph abstraction layer provides a unified interface for working with
provenance graphs, hiding the complexity of the underlying PROV representation
while enabling efficient graph algorithms through NetworkX integration.

\subsection{Node and Edge Abstractions}

The \texttt{GraphNode} class provides a consistent interface for working with
provenance elements regardless of their PROV type. Each node encapsulates an
identifier (the qualified name that uniquely identifies the element), a type
(entity, activity, or agent), and a dictionary of attributes. Methods provide
access to CPM-specific properties:
\texttt{is\_main\_activity()},
\texttt{is\_forward\_connector()},
\texttt{is\_backward\_connector()}, and
\texttt{is\_identifier\_entity()} check for CPM component types
by examining the \texttt{prov:type} attribute.

The base \texttt{GraphNode} class is extended by specialized variants for
different use cases. \texttt{DividedGraphNode} maintains separate storage
for traversal information attributes and domain-specific attributes. When
a document uses divided nodes, applications can access only TI attributes
for traversal purposes or only DS attributes for domain analysis, without
interference between the categories. \texttt{MergedGraphNode} provides a
unified view where all attributes are stored together, suitable for simpler
scenarios where TI/DS separation is not needed at the node level.

Similarly, \texttt{GraphEdge} represents relationships between nodes
with a consistent interface. Each edge has a source identifier, target
identifier, relation type (used, wasGeneratedBy, wasDerivedFrom, etc.),
and optional attributes. The \texttt{EdgeBuilder} class provides a
fluent interface for constructing edges:

\texttt{EdgeBuilder().from\_node('ex:a')}\\
\texttt{~~.to\_node('ex:b').with\_type('used').build()}

\noindent The \texttt{EdgeFilter} class encapsulates filter criteria for
edge queries, supporting predicates on source, target, type, and
attributes.

\subsection{ProvGraphWrapper}

The \texttt{ProvGraphWrapper} class is the central component of the graph
abstraction layer. It wraps a PROV document and provides graph operations
through a NetworkX-based representation. When constructed, the wrapper
analyzes the PROV document, builds a NetworkX directed graph with nodes
for each PROV element and edges for each PROV relation, and maintains
the mapping between PROV elements and graph nodes.

This dual representation—maintaining both the PROV document and the NetworkX
graph—is key to the design. The PROV document remains the authoritative
source for provenance content, ensuring full PROV compliance and enabling
serialization to standard formats. The NetworkX graph provides efficient
implementations of graph algorithms without reimplementing them. Changes
to one representation are reflected in the other to maintain consistency.

The wrapper provides methods for node and edge access that return
\texttt{GraphNode} and \texttt{GraphEdge} instances, shielding users
from PROV implementation details. Graph algorithms delegate to
NetworkX: \texttt{find\_paths} uses its path-finding algorithms,
\texttt{get\_predecessors} and \texttt{get\_successors} use its
neighbor queries, and connectivity analysis uses its component
algorithms.

Subgraph creation through \texttt{create\_subgraph} is particularly important
for CPM workflows. The method accepts a node filter predicate and returns a
new wrapper containing only matching nodes and their connecting edges. This
enables operations like extracting only traversal information (for cross-chain
navigation), extracting only domain-specific content (for detailed analysis),
or extracting the neighborhood of a specific node (for focused investigation).

\subsection{Factory Pattern}

Factory classes encapsulate creation logic for different component
strategies. \texttt{DividedCpmFactory} creates documents and nodes
that maintain separate TI and DS components, suitable for applications
that need to process these categories differently.
\texttt{MergedCpmFactory} creates simpler representations where all
attributes are stored together. Both factories implement a common
interface, allowing users to switch between strategies through
configuration without changing their code structure.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Template System}
\label{sec:template-system}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

The template system enables declarative specification of CPM documents and
provides interoperability with the Java reference implementation through
JSON serialization. This system is critical for workflows where documents
are defined in configuration files, exchanged between systems, or stored
for later reconstruction.

\subsection{Template Data Structures}

Templates are defined using Python dataclasses~\cite{pep557}, which provide automatic
generation of initialization, representation, and comparison methods while
reducing boilerplate code. The dataclass decorator generates \texttt{\_\_init\_\_},
\texttt{\_\_repr\_\_}, and \texttt{\_\_eq\_\_} methods based on field declarations,
enabling concise template definitions.

The primary class is \texttt{TraversalInformationTemplate}, which captures all
information needed to construct a complete CPM document. Its fields include:

\begin{lstlisting}
@dataclass
class TraversalInformationTemplate:
    bundle_name: str
    main_activity: MainActivityTemplate
    forward_connectors: List[ConnectorTemplate] = field(default_factory=list)
    backward_connectors: List[ConnectorTemplate] = field(default_factory=list)
    agents: List[AgentTemplate] = field(default_factory=list)
    identifier_entities: List[IdentifierEntityTemplate] = field(default_factory=list)
    namespaces: Dict[str, str] = field(default_factory=dict)
\end{lstlisting}

The \texttt{bundle\_name} identifies the provenance record with a qualified name.
The \texttt{main\_activity} field contains a \texttt{MainActivityTemplate}
describing the central activity—its identifier, optional label, and optional
start and end times. The connector lists contain \texttt{ConnectorTemplate}
instances capturing connector identifiers, labels, and connected entities.
Agent and identifier entity lists follow similar patterns. The namespaces
dictionary maps prefixes to URIs for identifier resolution.

Each component template captures identifier, optional label, and type-specific
attributes. Optional fields use Python's \texttt{Optional} type annotation and
default to \texttt{None}. This design supports both minimal templates (only
required fields) and fully specified templates (all optional fields populated).

\subsection{Serialization and Deserialization}

The \texttt{TraversalInformationSerializer} converts templates to JSON format
compatible with the Java reference implementation. Compatibility requires
exact field name matching, consistent date-time formatting, and proper handling
of optional fields.

The serializer uses Python's \texttt{json} module with custom encoding for
non-primitive types. Date-time values are formatted using ISO 8601 format
(\texttt{2024-01-15T10:00:00}) matching the Java implementation's format.
Qualified names are serialized as strings with their namespace prefix.
Optional fields are handled carefully—absent values are omitted from the
JSON output rather than included as null, producing cleaner output and
matching reference behavior.

The \texttt{TraversalInformationDeserializer} performs the reverse operation,
parsing JSON and constructing template instances. The deserializer validates
the JSON structure during parsing, checking for required fields, correct types,
and valid values. Malformed input produces clear error messages identifying
the specific problem—for example, "missing required field 'bundleName'" or
"invalid date format for startTime: expected ISO 8601".

\subsection{Template-Document Conversion}

The \texttt{TemplateProvMapper} converts between templates and PROV documents.
Converting a template to a document involves creating PROV elements for each
template component, establishing the required relationships between them,
and applying correct CPM type annotations.

The conversion process follows a defined sequence. First, the PROV document
is created with namespace bindings from the template. Then the Main Activity
is created with its identifier, label, and timestamps. Forward Connectors
are created with generation relationships to their connected entities and
the entities' usage by the Main Activity. Backward Connectors are created
with the Main Activity generating entities that the connectors use. Agents
are created with association relationships to the Main Activity. Identifier
Entities are created with appropriate relationships.

The reverse mapping extracts a template from an existing PROV document by
identifying CPM components through their type annotations and collecting
their attributes. This enables roundtrip conversion—a document can be
serialized to a template, stored or transmitted, deserialized, and
reconstructed without information loss.


%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{TI/DS Separation Algorithm}
\label{sec:ti-algorithm}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

The separation of traversal information from domain-specific content is
central to CPM's value proposition. The \texttt{TIAlgorithm} class implements
the classification logic, enabling applications to process TI and DS content
differently.

\subsection{Classification Rules}

The algorithm applies a hierarchy of rules to determine each component's
classification. The rules are applied in order, with earlier rules taking
precedence.

The primary rule examines type annotations. Components with explicit
CPM type annotations—such as \texttt{cpm:MainActivity},
\texttt{cpm:ForwardConnector}, \texttt{cpm:BackwardConnector},
or \texttt{cpm:IdentifierEntity}—are classified as traversal
information. This rule handles the core CPM components that form
the structural backbone of provenance chains.

Secondary rules handle entities connected to TI activities. When an entity
is generated by a Forward Connector (connected via \texttt{wasGeneratedBy}),
it serves as an input carrier for the Main Activity and is classified as TI.
When an entity is generated by the Main Activity and used by a Backward
Connector, it serves as an output carrier and is classified as TI. These
entities are structural—they exist to connect CPM activities rather than
to represent domain content.

Agents associated with TI activities through \texttt{wasAssociatedWith}
relations are classified as TI. These agents represent infrastructure
responsibility (the system performing the CPM activities) rather than
domain-specific accountability. This classification enables applications
to distinguish between system-level agents and domain-relevant agents.

All components not classified as TI through these rules are classified as
domain-specific. This conservative approach ensures that application content
is not accidentally hidden by infrastructure classification—when in doubt,
content is treated as domain-specific.

\subsection{Implementation Details}

The \texttt{TIAlgorithm} class is initialized with a \texttt{CpmDocument} or
\texttt{ProvGraphWrapper} instance. During initialization, it may optionally
precompute classifications for all components, or it may compute lazily on
first access.

The \texttt{belongs\_to\_traversal\_information} method accepts a PROV
element or \texttt{GraphNode} and returns a boolean indicating its
classification. The method first checks for explicit CPM type annotations,
then examines structural relationships to TI activities, and finally
defaults to DS classification.

Bulk query methods provide efficient access to classified components.
The \texttt{get\_traversal\_information\_nodes} method returns all TI
nodes in the document. The \texttt{get\_domain\_specific\_nodes} method
returns all DS nodes. These methods are more efficient than calling
\texttt{belongs\_to\_traversal\_information} for each node when processing
the entire document.

The implementation caches classification results for efficiency. Component
classifications are computed once and stored in a dictionary keyed by
identifier. When the document is modified (nodes or edges added or removed),
the cache is invalidated to ensure correctness. This approach avoids
redundant computation during analysis workflows that query many nodes while
ensuring that modifications produce correct results.


%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Validation Framework}
\label{sec:validation-framework}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

The validation framework ensures CPM compliance through comprehensive rule
checking. Validation is essential for catching structural problems early,
ensuring interoperability with other CPM systems, and providing clear
feedback to help users correct their documents.

\subsection{Validation Architecture}

The \texttt{CpmValidator} class orchestrates validation by applying a
collection of validation rules to a document. Each rule is implemented
as a separate class with a \texttt{validate} method that checks a specific
aspect of CPM compliance. This modular design enables adding new rules
without modifying existing code and allows selective rule application
when needed.

The validator aggregates results from all rules and categorizes them by
severity. \textbf{Errors} indicate violations that make the document
non-compliant with CPM—these must be fixed for the document to be valid.
\textbf{Warnings} indicate potential issues that may indicate problems
but don't necessarily violate the specification—users should review these
but may choose to accept them. \textbf{Informational messages} provide
observations that may be useful for understanding the document but don't
indicate problems.

Each validation result includes a message describing the issue in clear,
actionable language and identifiers of affected components to help users
locate problems. For example: "Error: Document contains no Main Activity"
or "Warning: Entity 'ex:orphan' is not reachable from the Main Activity".

\subsection{Validation Rules}

Structural rules verify document composition. The Main Activity rule
checks that exactly one Main Activity exists—zero triggers an error
("Document contains no Main Activity"), multiple trigger an error
("Document contains multiple Main Activities: ex:main1, ex:main2").
Connector rules verify that Forward Connectors generate entities used
by the Main Activity and that Backward Connectors use entities generated
by the Main Activity. Type annotation rules verify that CPM components
have correct \texttt{prov:type} attributes with proper CPM namespace URIs.

Connectivity rules verify graph structure. The reachability rule checks
that all CPM components are connected to the Main Activity through some
path—orphaned components trigger warnings with identification of the
unreachable components. This rule uses the graph abstraction layer's
connectivity analysis, delegating to NetworkX's algorithm. Direction
rules verify that generation relations connect activities to entities
and usage relations connect activities to entities.

Consistency rules verify logical coherence. Temporal rules check that
activities' start times precede their end times and that the temporal
ordering of generation and usage is consistent. Type consistency rules
verify that relations connect appropriate element types—\texttt{wasGeneratedBy}
should connect an entity to an activity, not an activity to an activity.
Identifier uniqueness rules verify that no two elements share the same
identifier within a bundle.

\section{Builder Pattern Implementation}
\label{sec:builder-pattern}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

The \texttt{CpmDocumentBuilder} provides a fluent interface for
constructing CPM documents through method chaining. Construction
begins with \texttt{with\_namespace} for prefix declaration, followed

\texttt{with\_forward\_connector} and
\texttt{with\_backward\_connector},
agents, and identifier entities. The \texttt{build} method validates
and returns the completed document, raising exceptions for missing
required components.

The builder handles default values and automatic relationship creation,
reducing boilerplate while allowing precise control when needed.

\section{Error Handling}
\label{sec:error-handling}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

The implementation defines custom exceptions:
\texttt{CpmException} as the base class, with
\texttt{CpmValidationError} for validation failures,
\texttt{CpmStructureError} for structural problems, and
\texttt{CpmConversionError} for serialization failures.
This hierarchy enables catching at appropriate specificity levels.

Public methods validate inputs immediately with specific messages like
"identifier 'ex:123' contains invalid characters" rather than generic
errors, helping users correct problems quickly.

\section{Getting Started}

The library can be installed using pip from the Python Package Index or

\end{lstlisting}

The library requires Python 3.8 or newer. All dependencies are automatically
installed during the pip installation process.

Creating a CPM document begins with instantiating a \texttt{CpmDocument} that
wraps an underlying PROV document. The PROV document is created using the
standard \texttt{prov} library, and the CPM document adds CPM-specific
functionality on top:

\begin{lstlisting}
from prov.model import ProvDocument
from src.cpm.model import CpmDocument

# Create underlying PROV document with namespace bindings
prov_doc = ProvDocument()
prov_doc.add_namespace('ex', 'http://example.org/')
prov_doc.add_namespace('cpm', 'https://2025.2.2.2/')

# Create CPM document wrapper
cpm_doc = CpmDocument(prov_doc)

# Add the Main Activity (required for CPM compliance)
cpm_doc.add_node(
    identifier='ex:main_activity',
    node_type='activity',
    attributes={
        'prov:type': 'cpm:MainActivity',
        'prov:label': 'Data Processing Pipeline'
    }
)

# Add input entity
cpm_doc.add_node(
    identifier='ex:input_data',
    node_type='entity',
    attributes={'prov:label': 'Raw Input Dataset'}
)

# Add output entity
cpm_doc.add_node(
    identifier='ex:output_data',
    node_type='entity',
    attributes={'prov:label': 'Processed Results'}
)

# Connect Main Activity to entities
cpm_doc.add_edge(
    source='ex:main_activity',
    target='ex:input_data',
    relation_type='used'
)
cpm_doc.add_edge(
    source='ex:output_data',
    target='ex:main_activity',
    relation_type='wasGeneratedBy'
)
\end{lstlisting}

This low-level approach provides full control over document construction but
requires understanding of CPM structure. The following sections demonstrate
higher-level approaches that simplify common patterns.


%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Using the Builder Pattern}
\label{sec:using-builder}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

The \texttt{CpmDocumentBuilder} provides a more convenient way to construct
CPM documents using a fluent interface. The builder guides users through the
construction process, handles default values, and validates the result:

\begin{lstlisting}
from src.cpm.builder import CpmDocumentBuilder
from datetime import datetime

# Build a complete CPM document using the fluent interface
doc = (CpmDocumentBuilder()
    # Declare namespaces
    .with_namespace('ex', 'http://example.org/')
    .with_namespace('custom', 'http://example.org/custom/')
    
    # Set bundle name
    .with_bundle_name('ex:data_processing_record')
    
    # Add the Main Activity with timing information
    .with_main_activity(
        identifier='ex:processing',
        label='Data Processing Pipeline',
        start_time='2025-01-15T10:00:00',
        end_time='2025-01-15T11:30:00'
    )
    
    # Add Forward Connector for input handling
    .with_forward_connector(
        identifier='ex:input_loader',
        label='Input Data Loader',
        generated_entity='ex:input_data'
    )
    
    # Add Backward Connector for output handling
    .with_backward_connector(
        identifier='ex:result_publisher',
        label='Result Publisher',
        used_entity='ex:output_data'
    )
    
    # Add responsible agent
    .with_agent(
        identifier='ex:system',
        label='Processing System v2.0',
        agent_type='prov:SoftwareAgent'
    )
    
    # Add identifier entity for cross-referencing
    .with_identifier_entity(
        identifier='ex:record_doi',
        label='DOI Reference',
        value='10.1234/example.2025.001'
    )
    
    # Build and validate the document
    .build())
\end{lstlisting}

The builder validates the document structure during the \texttt{build()} call.
If required components are missing—for example, if no Main Activity was
specified—the builder raises a descriptive exception. The builder also
automatically creates intermediate entities and establishes the required
relationships between components.

For simpler use cases, the builder supports minimal specification:

\begin{lstlisting}

    .with_namespace('ex', 'http://example.org/')

    .build())
\end{lstlisting}

\section{Graph Operations}

The library provides comprehensive graph traversal and analysis capabilities

successors = cpm_doc.get_successors('ex:main_activity')

\section{Use Case: BBMRI Biobank Data Transformation}
\label{sec:usecase-bbmri}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

This section demonstrates a complete real-world use case: transforming biobank
data from the BBMRI-ERIC~\cite{bbmrieric} (Biobanking and BioMolecular resources Research
Infrastructure) into CPM representation. This use case addresses the thesis
requirement to demonstrate transformation of information from existing formats
into the standardized representation defined by CPM.

\subsection{Use Case Context}

BBMRI biobanks track biological samples (tissue, blood, serum) through their
complete lifecycle. The data originates from multiple hospital and laboratory
systems in various formats---patient records from hospital databases, sample
records from Laboratory Information Systems, and storage logs from biobank
management software. This heterogeneous data must be transformed into a
standardized provenance representation that enables:

\begin{itemize}
\item Tracking sample origin and chain of custody
\item Linking provenance across organizational boundaries
\item Querying sample history for research and compliance
\item Interoperability with other biobanks using CPM
\end{itemize}

The use case is based on real BBMRI data structures from the MMCI (Masaryk
Memorial Cancer Institute) biobank, which is part of the Java CPM implementation
test data, ensuring compatibility verification.

\subsection{Source Data Structures}

The transformation begins with raw data records representing information
as it exists in hospital and biobank systems:

\begin{lstlisting}
# Patient record (from hospital database)
@dataclass
class PatientRecord:
    patient_id: str      # Hospital patient identifier
    sex: str             # Patient sex
    birth_year: int      # Year of birth
    gave_consent: bool   # Consent for sample use

# Sample record (from laboratory system)
@dataclass
class SampleRecord:
    sample_id: str       # Unique sample identifier
    patient_id: str      # Link to patient
    material_type: str   # S=Serum, P=Plasma, T=Tissue
    storage_type: str    # STS=Standard, LTS=Long Term
    diagnosis: str       # ICD-10 diagnosis code
    taking_date: datetime
    hospital_id: str     # Originating hospital

# Storage operation (from biobank logs)
@dataclass
class StorageRecord:
    storage_id: str
    sample_id: str
    storage_location: str
    stored_at: datetime
    operator_id: str
\end{lstlisting}

These data classes represent the ``existing format''---the way information
is currently stored in hospital and biobank systems before transformation.

\subsection{Transformation to CPM}

The transformation creates a CPM-compliant PROV document with proper
component structure. The key insight is mapping biobank concepts to CPM
components:

\begin{itemize}
\item The \textbf{storage operation} becomes the Main Activity
\item \textbf{Backward Connector} links to the sample acquisition bundle
\item \textbf{Forward Connector} enables linking to future distribution
\item \textbf{Domain-specific entities} capture sample and patient details
\end{itemize}

\begin{lstlisting}
def transform_to_cpm(patients, samples, storage_records):
    """Transform raw biobank records to CPM document."""
    doc = ProvDocument()
    
    # Define namespaces (matching Java implementation)
    bbmri = Namespace('bbmri', 
        'http://www.bbmri.cz/schemas/biobank/data#')
    cpm = Namespace('cpm', 
        'https://www.commonprovenancemodel.org/cpm-namespace-v1-0/')
    
    doc.add_namespace(bbmri)
    doc.add_namespace(cpm)
    
    # Create bundle for this storage operation
    bundle = doc.bundle(bbmri['storageBundle-33-BBM-2032-136043'])
    
    for sample in samples:
        # === MAIN ACTIVITY (CPM requirement) ===
        main_activity = bundle.activity(
            bbmri[f'storage-{sample.sample_id}'],
            other_attributes={
                'prov:type': cpm['mainActivity']
            }
        )
        
        # === BACKWARD CONNECTOR (link to acquisition) ===
        backward_conn = bundle.entity(
            bbmri[f'sampleAcqConnector-{sample.sample_id}'],
            other_attributes={
                'prov:type': cpm['backwardConnector'],
                cpm['referencedBundleId']: 
                    bbmri[f'acquisitionBundle-{sample.sample_id}']
            }
        )
        
        # === FORWARD CONNECTOR (link to distribution) ===
        forward_conn = bundle.entity(
            bbmri[f'sampleStorConnector-{sample.sample_id}'],
            other_attributes={
                'prov:type': cpm['forwardConnector']
            }
        )
        
        # === CPM RELATIONS ===
        bundle.usage(main_activity, backward_conn)
        bundle.generation(forward_conn, main_activity)
        bundle.derivation(forward_conn, backward_conn)
        
        # === DOMAIN-SPECIFIC: Sample Entity ===
        sample_entity = bundle.entity(
            bbmri[f'sample-{sample.sample_id}'],
            other_attributes={
                'prov:type': 'pbm:sample',
                bbmri['sampleId']: sample.sample_id,
                bbmri['materialType']: sample.material_type,
                bbmri['diagnosis']: sample.diagnosis,
                bbmri['takingDate']: sample.taking_date.isoformat()
            }
        )
        bundle.specialization(sample_entity, forward_conn)
    
    return CpmDocument(doc)
\end{lstlisting}

\subsection{Querying Transformed Data}

# TI/DS separation for selective processing

\section{Test Suite Organization}
\label{sec:test-suite-overview}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

The test suite is organized to mirror the source code structure, making it
easy to locate tests for specific functionality and ensuring comprehensive
coverage of all components.

The \texttt{tests/model/} directory contains tests for the \texttt{CpmDocument}
class and its mixins. Test files are organized by functionality:
\begin{itemize}
\item \texttt{test\_cpm\_document\_constructor.py} ---
  document creation and initialization
\item \texttt{test\_cpm\_document\_modification.py} ---
  CRUD operations on nodes and edges
\item \texttt{test\_cpm\_document\_removal.py} ---
  deletion operations and edge handling
\item \texttt{test\_cpm\_document\_equals.py} ---
  equality comparison
\item \texttt{test\_cpm\_utilities.py} ---
  helper methods and utilities
\end{itemize}
This fine-grained organization improves maintainability
and makes test failures easy to diagnose.

The \texttt{tests/graph/} directory tests graph abstraction components:
\texttt{test\_node.py} for \texttt{GraphNode} and its variants,
\texttt{test\_edge.py} for \texttt{GraphEdge} and edge filtering,
\texttt{test\_wrapper.py} for \texttt{ProvGraphWrapper} operations,
and \texttt{test\_factory.py} for factory classes.

Root-level test files cover cross-cutting concerns: \texttt{test\_template.py}
tests the template system including serialization, deserialization, and
conversion; \texttt{test\_validation.py} tests the validation framework
including all rule categories; \texttt{test\_advanced\_traversal.py} tests
complex graph traversal scenarios; and \texttt{test\_adapters.py} tests
PROV integration utilities.

The suite includes several categories of tests. \textbf{Unit tests} verify
individual methods in isolation, using mocks to control dependencies and
test specific behaviors. \textbf{Integration tests} verify that components
work together correctly, such as template-to-document conversion or validation
of builder-created documents. \textbf{Parity tests} verify compatibility
with the Java reference implementation using known inputs and expected outputs.
\textbf{Edge case tests} verify handling of boundary conditions, empty inputs,
maximum sizes, and error conditions.

Tests are executed using pytest~\cite{pytest} with the command:
\texttt{pytest tests/ --cov=src --cov-report=html}

This generates both console output and an HTML coverage report for detailed
analysis of tested and untested code paths.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Reference Implementation Parity}
\label{sec:reference-parity}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

Parity tests are critical for ensuring that the Python implementation is
compatible with the Java reference implementation. These tests use known
inputs and expected outputs from the Java implementation to verify that
the Python implementation produces equivalent results.

\textbf{Template serialization parity} verifies that JSON output from the
Python serializer matches the Java format exactly. Test cases use templates
with various component combinations, serialize them to JSON using both
implementations, and compare the results. The comparison accounts for
field ordering (which may differ between implementations) while verifying
that all content matches exactly.

\textbf{TI/DS classification parity} verifies that component classification
produces identical results. Test cases use documents with known classifications
from the Java implementation and verify that the Python \texttt{TIAlgorithm}
produces the same decisions. This is particularly important for complex
documents where classification depends on relationship analysis.

\textbf{Validation parity} verifies that validation rules are applied
consistently. Test cases include both valid and invalid documents with known
validation results from Java. The tests verify that the same errors and
warnings are produced, with equivalent messages identifying the same
problematic components.

The parity test suite is organized in
\texttt{tests/model/} under the file
\texttt{test\_java\_parity\_comprehensive.py}
and includes 25 carefully designed test cases covering all major
functionality areas. When discrepancies are discovered, they are investigated to determine
whether the Python implementation needs correction or whether the difference
represents an intentional deviation (documented and justified).

\section{Performance Evaluation}
\label{sec:performance-evaluation}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

Performance was measured to verify that the implementation meets the efficiency
requirements established in Chapter~\ref{chap:analysis-requirements}. Measurements
used Python's \texttt{time.perf\_counter()} for high-resolution timing, with
each operation repeated 100 times and results averaged to reduce variance.
Tests were conducted on a standard development workstation with Python 3.11.

\textbf{Document creation performance} scales linearly with document size as
expected. Creating a document with 10 nodes completes in approximately 1.2ms,
100 nodes in 9.3ms, and 1000 nodes in 92ms. This linear scaling indicates
that the implementation avoids algorithmic inefficiencies that would cause
superlinear growth.

\textbf{Graph operation performance} benefits from NetworkX's optimized
algorithms:

\begin{table}[h]
\centering
\begin{tabular}{|l|r|r|r|}
\hline
\textbf{Operation} & \textbf{100 nodes} & \textbf{500 nodes} & \textbf{1000 nodes} \\
\hline
get\_predecessors & 0.1 ms & 0.2 ms & 0.3 ms \\
get\_successors & 0.1 ms & 0.2 ms & 0.3 ms \\
find\_paths & 0.5 ms & 2.1 ms & 4.8 ms \\
traverse\_from & 0.3 ms & 1.2 ms & 2.5 ms \\
create\_subgraph & 0.8 ms & 3.5 ms & 7.2 ms \\
validate & 1.2 ms & 5.8 ms & 12.3 ms \\
\hline
\end{tabular}
\caption{Graph operation performance by document size}
\label{tab:operation-performance}
\end{table}

Predecessor and successor queries complete in sub-millisecond time even for
large documents. Path finding scales with graph complexity but remains under
5ms for 1000-node graphs with typical connectivity. Validation, which must
check multiple rules across the entire document, takes longer but still
completes in under 15ms for large documents.

\textbf{Template serialization performance} also scales linearly. JSON
serialization completes in 0.8ms for small documents (10 nodes) and 6.2ms
for larger documents (100 nodes). Deserialization is slightly slower due
to validation during parsing but remains in the same order of magnitude.

\textbf{Memory usage} was measured by comparing process memory before and
after document creation. The implementation maintains approximately 1.6x
overhead over the base PROV document size, well within the 2x target. The
additional memory supports the NetworkX graph representation for efficient
algorithms and cached classification results for performance.

\chapter{Conclusion}
\label{chap:conclusion}

This thesis presented the design, implementation, and evaluation of a Python
library for the Common Provenance Model (CPM). The work addressed a significant
gap in the Python ecosystem: while Java developers had access to a mature,
well-tested CPM implementation, Python developers—who dominate in data science,
scientific computing, and machine learning—lacked native CPM support. This
thesis fills that gap with a comprehensive library that provides equivalent
functionality to the Java reference implementation while following Python
conventions and integrating with the Python ecosystem.

The implementation successfully delivers all major components required for
CPM support in Python. The CpmDocument class serves as the primary user
interface, providing comprehensive document manipulation through a well-organized
mixin architecture that separates concerns into CoreMixin for CRUD operations,
IOMixin for serialization, AnalysisMixin for statistics and validation, and
TraversalMixin for graph navigation. The template system enables declarative
document specification with JSON serialization fully compatible with the Java
reference implementation, enabling seamless provenance exchange between Python
and Java systems. The TI/DS separation algorithm correctly classifies components
as traversal information or domain-specific content, matching the reference
implementation behavior as verified through parity tests. Graph operations
through NetworkX integration provide efficient implementations of traversal,
path finding, and subgraph extraction. Comprehensive validation ensures CPM
compliance through a modular rule-based framework with clear error messages.

The test suite demonstrates the implementation's reliability with over 305
test cases achieving 92\% code coverage. Parity tests specifically verify
compatibility with the Java reference implementation for template serialization,
component classification, and validation results. The BBMRI biobank use case
demonstrates the practical applicability of the library, showing a complete
transformation pipeline from existing formats (hospital and biobank database
records) into standardized CPM representation.

While the implementation achieves its goals, several limitations should be
acknowledged. The library handles only static documents with no support for
streaming provenance. Persistence is file-based only, with no database backend.
Scale limits exist for very large graphs with millions of nodes. There is no
automatic provenance capture—users must explicitly create provenance records.

Several directions for future work could extend the library's capabilities.
Workflow engine integration with Apache Airflow or Prefect would enable
automatic provenance capture from pipeline execution. Visualization tools
could render provenance graphs for human inspection. Performance optimization
through lazy loading or database backends could extend scalability. An
asynchronous API would enable integration with modern async frameworks.
Extended validation with custom domain-specific rules would enable applications
to enforce domain requirements beyond CPM structural rules.

Data provenance is increasingly recognized as essential in data-driven
applications. Scientific reproducibility requires knowing exactly how results
were produced. Regulatory compliance demands audit trails for data processing.
Machine learning fairness and debugging require understanding training data
lineage. The Common Provenance Model provides a foundation for interoperable,
structured provenance that can span organizational boundaries and diverse
systems. This Python implementation makes CPM accessible to the large and
growing community of Python developers, researchers, and data scientists.
By providing native support that integrates with Python conventions and the
Python ecosystem, the library removes barriers to CPM adoption in Python-based
workflows. We hope this work contributes to wider adoption of structured
provenance practices in scientific computing, data engineering, and beyond.
